## 1. 核心架构：两层分离式退出体系

> [!IMPORTANT]
> **Layer 1（硬性防线）和 Layer 2（预警减仓）是完全独立的两套系统。**  
> Layer 2 的目标是赶在 Layer 1 触发之前提前走掉一半。  
> Layer 1 是不可协商的兜底断路器。

```
Layer 2（软预警）：筹码+情绪+动量+时间 → 提前减仓 50%
                                          ↓
                              k 从 2.8 收紧到 1.5
                                          ↓
Layer 1（硬防线）：价格 < Chandelier Stop → 清仓剩余（或全部）
```

## 2. Layer 1：硬性防线（即时执行，不走评分）

### 2.1 规则 1：ATR 破位止损

**算法**：Chandelier Exit

```
TR_i = max(High_i − Low_i, |High_i − Close_{i-1}|, |Low_i − Close_{i-1}|)
ATR_t = (11/12) × ATR_{t-1} + (1/12) × TR_t    (周期 n=12)
HH = max(入场以来所有日线最高价)
Stop = HH − k × ATR_t
```

**数据**：日线 OHLCV（盘前计算）+ L1 快照 last_price（盘中判定）

**参数**：

| 参数 | 值 | 依据 |
|:---|:---:|:---|
| ATR 周期 | 12 | 半月交易日 |
| 正常 k | 2.8 | A 股 ETF 盘中回撤 / ATR 比 ≈ 2.0-2.3×，留 25% 余量 |
| 减仓后 k | 1.5 | Layer 2 已减仓 50% 后进入保命模式 |

**触发后行为 — 根据软信号上下文和数据可用性决定力度**：

```
当 last_price < Stop 时：

  先检查数据健康状态（见第 3.3 节）

  IF  any(signal == UNAVAILABLE)（数据缺失）:
      → 卖出 sellable_qty（可用余额），不卖 T+1 锁定部分
      → 告警：「⚠️ 数据缺失：{列表}，已按最高风险执行」
      → 如有 T+1 锁定股无法卖出，追加告警

  IF  Score_soft > 0（有预警信号亮灯）:
      → 卖出 sellable_qty（可用余额）

  IF  Score_soft = 0  AND  本波段未使用过救生衣:
      → 70% 清仓（从 sellable_qty 中扣减）
      → 保留 30% 作为"救生衣仓位"
      → 对救生衣设置独立超紧止损：再跌 1% → 清掉
      → 记录卖出时间 sell_time
      → 回补条件（全部满足）：
          a) 已经过 30 个交易分钟（排除休市时间，见下注）
          b) last_price > Stop
          c) Score_soft = 0
          d) data_feed ≠ STALE
          e) last_price > 跌停价 × 1.02（过滤跌停开板死猫跳）
          f) 当前时间 ≤ 14:30（禁止尾盘回补，保留足够日内保护窗口）
      → 满足后买回 70%，挂 tick_ceil(Ask1 × 1.003)
          标记 lifeboat_used = True
          此时仓位构成：
            - 30% 旧股 = sellable（日内可卖）
            - 70% 新股 = T+1 锁定（日内不可卖）
          告警：「⚠️ 假洗盘回补完成。可卖余额 30%，
                 T+1 锁定 70%。日内仅 30% 有止损保护。」

  IF  Score_soft = 0  AND  lifeboat_used = True:
      → 卖出 sellable_qty（仅可用余额，可能只有 30% 旧股）
      → IF  total_qty > sellable_qty:
          告警：「⚠️ 尚有 {locked_qty} 股 T+1 锁定无法卖出，
                 明日开盘将自动挂卖」
          次日 09:30 自动卖出剩余锁定仓位

  所有 Layer 1 卖出挂单价 = max(跌停价, tick_floor(Bid1 × 0.98))
  所有 Layer 1 买入挂单价 = tick_ceil(Ask1 × 1.003)
```

> [!IMPORTANT]
> **持仓余额区分**：系统必须在每次操作前查询 `total_qty`（总持仓）和 `sellable_qty`（可用余额）。  
> `locked_qty = total_qty - sellable_qty` 为 T+1 锁定股，日内不可卖出。  
> 所有卖出指令只能操作 `sellable_qty`。

**救生衣机制的数学依据**：
- 最坏额外亏损：6 万 × 1% = **600 元**（总资金 0.3%）
- 若反弹回补后，70% 新股 T+1 裸露风险 ≈ 14 万 × 3% = **4200 元**（最坏）
- 赔率 3:1，反弹概率 ~40% > 盈亏平衡点 25%。**正期望。**

### 2.2 规则 2：跳空跌穿保护

```
检查时机：09:25 集合竞价结束 + 13:00 午盘复开

IF  当前快照价 < Stop × 0.97（涵盖早盘 09:30 和午盘 13:00）
THEN  立即卖出 sellable_qty，无救生衣
      挂单价 = max(跌停价, tick_floor(Bid1 × 0.98))
      IF locked_qty > 0:  告警「T+1 锁定 {locked_qty} 股次日自动卖出」
```

> [!NOTE]
> A 股 11:30-13:00 午休期间可能发生系统性利空，13:00 复盘跳空低开。必须覆盖。

### 2.3 规则 3：死水区强平

```
IF  days_held ≥ 10  AND  |current_return| ≤ 2%
THEN  全部清仓
      挂单价 = tick_floor(Bid1 × 0.98)
```

### 2.4 规则 4：T+0 日内熔断（仅限做 T 操作）

```
IF  当日 T+0 做 T 的已实现亏损 ≥ 0.3%（≈600 元）
THEN  暂停 T+0 做 T 操作至次日
      波段持仓的 Layer 1 / Layer 2 退出逻辑不受任何影响
```

> [!WARNING]
> 熔断只冻结 T+0 日内做 T。**波段止损指令永远可执行**，不受熔断限制。

### 2.5 动态 Chandelier 收紧（由 Layer 2 驱动）

| 条件 | k 值 | 说明 |
|:---|:---:|:---|
| 正常（S_chip < 0.3） | **2.8** | 标准 |
| 筹码初步恶化（S_chip ≥ 0.3） | **2.38**（×0.85） | 预防性收紧 |
| Layer 2 已减仓 50% | **1.5** | 保命模式 |

### 2.6 挂单价汇总

| 场景 | 挂单价 | 取整 | 理由 |
|:---|:---|:---:|:---|
| Layer 2 预警减仓 | Bid1 | floor | 非恐慌，正常流动性 |
| **所有 Layer 1 卖出** | **Bid1 × 0.98** | **floor** | **20 万资金成交优先 > 价格优势** |
| 救生衣回补买入 | Ask1 × 1.003 | ceil | 上浮确保买入成交 |

### 2.7 数量整手规则（新增硬约束）

> [!IMPORTANT]
> **买入必须整手（100 的整数倍），卖出允许零股。**

```
lot = 100
round_down_lot(q) = (q // lot) * lot
```

适用范围：
- Layer 2 预警减仓（卖出 50%）：为保持简洁与避免残留，卖出数量按整手取整（round_down_lot）
- 救生衣回补买入（买回 70%）：买入数量必须整手取整（round_down_lot）

不适用范围（必须允许零股卖出）：
- Layer 1 全清 / 跳空保护 / 死水区 / 救生衣卖出 70%：卖出允许零股，以确保“清仓意图”真正落地
- 次日 09:30 pending 自动挂卖：卖出允许零股，避免零股永远无法清理

**价格精度规则**：
- A 股 ETF 最小价格变动单位 = **0.001 元**
- 卖出：`tick_floor(price) = math.floor(price * 1000) / 1000`
- 买入：`tick_ceil(price) = math.ceil(price * 1000) / 1000`
- 示例：Bid1=1.234 × 0.98 = 1.20932 → `tick_floor` → **1.209**

## 3. Layer 2：预警减仓（四维融合评分）

### 3.1 评分公式

```
Score_soft = 0.7 × S_chip  +  0.7 × S_sentiment  +  0.5 × S_diverge  +  0.4 × S_time
```

**S_trail 不参与此评分。** Layer 1 和 Layer 2 完全独立。

| 信号 | 类型 | 满分贡献 | 计算时机 |
|:---|:---:|:---:|:---|
| S_chip | 连续 [0,1] | 0.7 | 盘前（日线级） |
| S_sentiment | 二值 0/1 | 0.7 | 盘前（LLM T+1） |
| S_diverge | 二值 0/1 | 0.5 | 盘前（日线级） |
| S_time | 二值 0/1 | 0.4 | 盘前（计数器） |

**理论满分 = 2.3**

### 3.2 阈值

| 阈值 | 值 | 动作 |
|:---|:---:|:---|
| 预警减仓 | **≥ 0.9** | 卖出 50%，k 收紧到 1.5，暂停做 T |

**典型触发组合验证**：

| 组合 | 得分 | 触发？ |
|:---|:---:|:---:|
| 仅 S_time | 0.4 | ❌ |
| S_chip(0.5) + S_time | 0.75 | ❌ |
| S_chip(0.5) + S_sentiment | **1.05** | ✅ |
| S_sentiment + S_diverge | **1.20** | ✅ |
| S_chip(0.5) + S_diverge + S_time | **1.25** | ✅ |

### 3.3 数据健康监控

每个信号维护三态：`{正常值, UNAVAILABLE}`

```
data_health = {
    "S_chip":      AVAILABLE / UNAVAILABLE,   # 筹码引擎崩溃或冷启动期
    "S_sentiment": AVAILABLE / UNAVAILABLE,   # LLM 超时/API 失败
    "S_diverge":   AVAILABLE / UNAVAILABLE,   # 日线数据拉取失败
    "S_time":      AVAILABLE / UNAVAILABLE,   # 纯计数器，极少缺失
}
```

**降级策略**：

| 场景 | 影响 | 处理 |
|:---|:---|:---|
| Layer 1 触发时有信号缺失 | 无法判定 Score_soft 是否真为 0 | **100% 清仓，无救生衣** |
| 日常评分时有信号缺失 | Score_soft 可能被低估 | 缺失信号贡献 = 权重 × 0.5 |
| 筹码引擎冷启动（< 10 天） | DPC 序列不可靠 | S_chip = UNAVAILABLE |

**所有 UNAVAILABLE 事件立即告警用户**：

```
「⚠️ 信号降级运行：{信号名} 数据不可用，原因：{具体原因}。
  Layer 1 已切换为最高风控模式（无救生衣），
  Layer 2 以中等风险假设运行。」
```

**外部数据时效性校验**（盘前 08:30）：

```
对每个外部数据源校验时间戳：
  DPC 数据日期 ≠ 上一个交易日 → S_chip = UNAVAILABLE
  LLM 评分日期 ≠ 上一个交易日 → S_sentiment = UNAVAILABLE
  日线 OHLCV 最新日期 ≠ 上一个交易日 → S_diverge = UNAVAILABLE（重试至 09:15）

告警：「⚠️ {数据源} 数据过期（最新：{日期}，期望：{上一交易日}），
       {信号名} 已降级为 UNAVAILABLE。请检查 {引擎/管线} 是否正常运行。」
```

**冷启动专用告警**（每日盘前，直到暖机完成）：

```
「⚠️ 筹码引擎冷启动第 {n}/10 天，S_chip 信号已禁用。
  仅依赖 S_sentiment + S_diverge + S_time + Layer 1 硬止损运行。」
```

## 4. 信号详述

### 4.1 S_chip — 渐进式筹码恶化 [0, 1]

**DPC 定义**：成本 ≤ 现价 × 0.85 的筹码占比  
**数据**：筹码引擎 `calc_deep_profit_ratio()` + `calc_profit_ratio()`，每日盘后更新  
**维护**：5 天滑动窗口 `[DPC_{t-4}, ..., DPC_t]`  
**暖机**：引擎运行 < 10 个交易日时，S_chip 强制 = UNAVAILABLE

**算法**：

```
前提：profit_ratio ≥ 75%（否则 S_chip = 0）

Tier 1 — 单日异动（Day 1 响应，+0.5）:
  daily_drop = (DPC_t − DPC_{t-1}) / DPC_{t-1}
  IF daily_drop ≤ −15%  THEN  score += 0.5

Tier 2 — 窗口内下降趋势（Day 2 响应，+0.3）:
  drop_days = 5 日窗口内 DPC 下降的天数（允许中间反弹不清零）
  IF drop_days ≥ 3  THEN  score += 0.3

Tier 3 — 累计确认（Day 3-5，+0.2）:
  total_drop = (DPC_t − DPC_{t-4}) / DPC_{t-4}
  IF total_drop ≤ −20% AND drop_days ≥ 3  THEN  score += 0.2

S_chip = min(score, 1.0)
```

**参数**：

| 参数 | 值 | 依据 |
|:---|:---:|:---|
| DPC 利润缓冲 α | 15% | 成本低于现价 15% = 机构铁底仓 |
| 前提获利盘 | 75% | < 75% 不算高位派发环境 |
| Tier 1 暴跌阈值 | −15% | DPC 日常噪音 ±5-8%，−15% ≈ 2σ |
| Tier 2 窗口内下降天数 | ≥3/5 天 | 允许 1 天反弹不清零，防锯齿型 DPC 绕过 |
| Tier 3 累计降幅 | −20% | 匹配引擎精度可检测下限 |
| 冷启动暖机 | 10 天 | MaxEnt 三角分布法需 10 日日线修正 |

### 4.2 S_sentiment — LLM 情绪预警 {0, 1}

**数据**：`finintel --signal-etf` 输出的 `sentiment_score ∈ [0,1]`  
**更新**：每日盘后，T+1 延迟

```
S_sentiment = 1  IF  sentiment_score ≤ 0.35
```

| 参数 | 值 | 依据 |
|:---|:---:|:---|
| 恶化阈值 | 0.35 | 日常噪音 0.5-0.7，0.35 = 结构性利空 |

### 4.3 S_diverge — 动量衰竭复合信号 {0, 1}

**数据**：日线 Close / High / Low / Volume（至少 30 日回看）

```
S_diverge = 1  当且仅当：
  RSI 顶背离已确认  AND  (ADX>35 拐头  OR  缩量新高 ≥ 2 次)
```

**子信号 A — RSI 顶背离**：

```
RSI = Wilder EMA(14) 标准算法
波峰检测："前3后3"滚动窗口局部极大值
找最近两个价格波峰 P1, P2（间距 ≥ 5 天）

触发：P2 > P1  AND  RSI_2 < RSI_1  AND  RSI_1 > 65
确认滞后：3 天（波峰需右侧数据验证）
```

**子信号 B — ADX 拐头**：

```
ADX = 标准 DMI 算法，周期 14
触发：ADX_t > 35  AND  ADX_t < ADX_{t-1}  AND  ADX_{t-1} ≥ ADX_{t-2}
```

**子信号 C — 缩量新高**：

```
近 8 日窗口内，每个新高日（Close_d > 近 5 日 Close）：
  IF Volume_d < MA5_Volume × 0.8  THEN 记一次
IF 次数 ≥ 2  THEN 子信号C = True
```

**参数**：

| 参数 | 值 | 依据 |
|:---|:---:|:---|
| RSI 周期 | 14 | 标准 Wilder |
| 超买前提 | 65 | A 股 ETF 强趋势中 RSI 在 65-75 |
| ADX 强趋势阈值 | 35 | A 股需上调（美股经验值 25 太低） |
| 缩量判定 | < MA5×0.8 | 突破日量能萎缩 20% |

### 4.4 S_time — 时间停滞 {0, 1}

**数据**：days_held、days_since_high（纯计数器）

```
S_time = 1  当且仅当：
  days_held > 15  AND  days_since_high ≥ 6  AND  current_return ≥ 5%
```

| 参数 | 值 | 依据 |
|:---|:---:|:---|
| 最低持仓 | 15 天 | 主升浪均值 12-22 天 |
| 滞涨天数 | 6 天 | 超过 1 周无新高 |
| 浮盈前提 | 5% | 亏损头寸走死水区规则 |

## 5. 两阶段退出执行流

### Phase 1：Layer 2 预警减仓（Score_soft ≥ 0.9）

| 步骤 | 动作 |
|:---|:---|
| 1 | 写入状态文件：`{"intent": "REDUCE_50", "status": "PENDING"}` |
| 2 | 卖出 50%（取整到 100 份），挂 Bid1 |
| 3 | 查询实际持仓确认成交 |
| 4 | 写入状态文件：`{"status": "DONE", "actual_sold": N}` |
| 5 | k 收紧 2.8 → 1.5，重算 Stop |
| 6 | 标记 `reduced = True`，暂停 T+0 做 T |

### Phase 2：Layer 1 硬止损（price < Stop）

| 步骤 | 动作 |
|:---|:---|
| 1 | 写入状态文件：`{"intent": "HARD_STOP", "status": "PENDING"}` |
| 2 | 根据 Score_soft / data_health / lifeboat_used 决定 100% 或 70%+30% |
| 3 | 执行卖出，挂 max(跌停价, tick_floor(Bid1 × 0.98)) |
| 4 | 通过券商委托列表查询订单号 + 状态确认成交 |
| 5 | 写入状态文件：`{"status": "DONE"}` |
| 6 | 如有救生衣，设置超紧止损 |

### 崩溃恢复（Write-Ahead State）

```
状态文件写入（原子操作）：
  1. 将新状态写入临时文件 state.json.tmp
  2. 调用 os.replace("state.json.tmp", "state.json")  ← NTFS 上原子操作
  3. 永远不直接 open("state.json", "w") 写入

系统启动时：
  1. 读取 state.json（如果 .tmp 存在但 .json 不存在 → 上次写入中断 → 告警）
  2. 查询 easytrader 实际持仓：total_qty + sellable_qty
  3. 查询当日未成交委托（在途订单），避免重复提交
  4. 对比：
     存在在途卖单               → 等待其成交/撤单后再决策
     实际持仓 ≈ state 记录     → 正常恢复
     实际持仓 < state 记录     → 卖单已成交，补更新 state
     实际持仓 > state 记录     → 不应出现 → 告警人工介入
     state.json 不存在         → 以实际持仓为准，告警

  核心原则：真相来源 = 券商持仓 + 当日委托，不是本地变量。
```

## 6. 场景走查

### 场景 1：教科书式慢顶

```
Day 0:  入场 新能源 ETF 1.000
Day 12: 涨到 1.150（最高），RSI=71
Day 16: RSI 背离确认 + DPC 连续下降 → S_chip=0.5, S_diverge=1
        Score_soft = 0.35 + 0.5 = 0.85 → HOLD（差一点）
        S_chip ≥ 0.3 → k 预收紧 2.8→2.38
Day 20: S_time 触发 → Score_soft = 1.25 ≥ 0.9 → ★ REDUCE_50 ★
```

## 8. 生产加固

### 8.1 🔴 P0 — easytrader 会话断连

```
提交卖/买单后：
  “委托确认” = 查询券商委托列表中存在 order_id + 状态为已报/部分成交/已成
  （不依赖 GUI 弹窗通知，必须通过委托查询接口确认）
  必须在 10 秒内获得委托确认
  IF 超时或 Exception：
    → 重试 1 次
    → 仍然失败 → 系统进入“冻结模式”：
      停止所有新主动交易决策
      但不阻止已持久化的 pending 卖出（pending_sell_locked / pending_sell）
      每 30 秒重试卖出
      紧急告警：「🚨 easytrader 断连，挂单未提交，请立即人工介入」
```

### 8.2 🔴 P0 — 跌停板无买盘

```
IF  Bid1_vol = 0  OR  last_price ≤ 跌停价：
  → 挂跌停价排队卖出
  → 告警：「⚠️ 疑似跌停，卖单已排队，可能无法当日成交」
  → 每 30 秒检查成交状态
  → 14:55 仍未成交 → 告警升级：「🚨 卖单当日未成交，次日开盘将自动重挂」
  → 写入 state.json pending_sell
```

### 8.3 🔴 P0 — Layer 1/2 执行竞态

```
执行锁（Mutex）：
  同一时刻只允许一个卖出/买入操作在执行
  Layer 1 优先级 > Layer 2

  IF Layer 1 触发时 Layer 2 正在执行：
    → 等待 Layer 2 完成（最多 10 秒）
    → 查询实际 sellable_qty
    → 基于最新持仓执行 Layer 1

  IF Layer 2 触发时 Layer 1 正在执行：
    → 取消 Layer 2（Layer 1 已经在清仓，无需再减仓）
```

### 8.4 🟡 P1 — L1 快照数据停滞

```
记录每个快照的时间戳
IF  当前系统时间 − 最新快照时间 > 15 秒：
  → 标记 data_feed = STALE
  → 暂停所有基于价格的决策（Layer 1 不触发、救生衣不判定）
  → 告警：「⚠️ L1 数据已停滞 {N} 秒，暂停交易决策」
  → 尝试重连 XtQuant
```

### 8.5 🟡 P1 — DPC 除零保护

```
Tier 1 计算时：
  IF  DPC_{t-1} < 0.01：
    → 跳过 Tier 1（无法计算有意义的百分比变动）
    → S_chip 仅依赖 Tier 2 / Tier 3

Tier 3 计算时：
  IF  DPC_{t-4} < 0.01：
    → 跳过 Tier 3
```

### 8.6 🟡 P1 — 部分成交

```
提交卖单后查询成交：
  IF  成交数量 < 委托数量：
    remaining_qty = 委托数量 − 成交数量

    IF  remaining_qty < 100（不足一手）：
      → 停止重试，记录为"残余持仓"
      → 告警：「ℹ️ 剩余 {qty} 份未成交（≈{金额}元），
               不足一手，已放弃重试。可手动处理。」
      → 不视为错误，不进入冻结模式

    IF  remaining_qty ≥ 100：
      → 等待 5 秒 → 撤销未成交部分 → 以更激进价格重挂
      → 最多重试 3 次
      → 3 次后仍有 ≥ 100 份未成交 → 告警人工介入
```

### 8.7 🟢 P2 — 防止双实例

```
启动时创建 .exit_manager.lock
IF lock file 已存在 且 进程仍在运行：
  → 拒绝启动，告警「⚠️ 已有实例在运行」
IF lock file 存在但进程已死：
  → 清除旧 lock，正常启动
```

### 8.8 🟢 P2 — 跨假期 state 持久化

```
state.json 中的 pending_sell_locked 不依赖具体日期
启动时只检查“是否有 pending” → 如有则执行
即使间隔春节/国庆等长假也能正确恢复

同理：跌停未成交的 pending_sell 也持久化
次日（或下一个交易日）开盘后自动重挂
```

### 8.9 🟢 P2 — 盘前日线数据未就绪

```
08:30 拉取日线时，校验最新日期：
IF  最新日期 ≠ 上一个交易日：
  → 每 60 秒重试，最多到 09:15
  → 09:15 仍未就绪 → S_diverge = UNAVAILABLE
  → 告警：「⚠️ 日线数据未更新至昨日，ATR/RSI/ADX 可能不准」
  → ATR/Stop 沿用前日的值（保守处理）
```

### 8.10 T+1 锁定仓位次日处理

```
当 Layer 1 触发时存在 locked_qty > 0：
  → 写入 state.json: {"pending_sell_locked": qty}
  → 次日（或下一个交易日）盘前检查此字段
  → 09:30 无条件挂卖（不重新评估）
  → 理由：趋势假设已被两次价格行为证伪，T+1 只是技术延迟，意图不变
```

### 8.11 多 pending 并存合并规则

```
当多个 pending 同时存在时（pending_sell_locked + pending_sell）：
  1. 合并目标数量：总待卖 = sum(各 pending 的 qty)
  2. 去重：待卖 ≤ total_qty（不可超卖）
  3. 优先级：pending_sell_locked（T+1 解锁）先执行，然后 pending_sell
  4. 执行后更新所有相关 pending 状态
  5. 每条 pending 有唯一 id，防止重复执行
```

### 8.12 冷却期计时规则

```
冷却期 = 30 个交易分钟（排除休市时间）

示例：
  11:20 卖出 → 交易分钟计10 分钟（11:20-11:30）
  11:30-13:00 休市 → 不计时
  13:00 复盘 → 继续计时20 分钟（13:00-13:20）
  13:20 冷却期满 → 开始检查回补条件
```
