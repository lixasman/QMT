## 2. 核心架构：三层分级仓位管理体系

> [!IMPORTANT]
> 仓位管理分为三个独立层级，每一层有独立的状态空间和触发权限。
> **Layer A（组合级）→ Layer B（头寸级）→ Layer C（日内级）**

```
Layer A: 账户与组合管理
  ┌─ 账户熔断器（10% HWM Drawdown → 强制全清）
  ├─ 相关性过滤器（ρ ≥ 0.60 → 互斥）
  ├─ ATR 风险平价仓位计算器（动态定量，上限7万）
  └─ 资金分配器（2 Slot × 7万 + 6万 Reserve）
                    ↓
Layer B: 单标的头寸生命周期（本文核心）
  ┌─ 入场阶段：试探 → 确认 → 底仓建立
  ├─ 加仓阶段：Scale_1 → Scale_2（金字塔递减）
  ├─ 退出阶段：Layer 2 预警减仓 → Layer 1 硬止损
  └─ 状态重置：减仓后回补视为新建仓
                    ↓
Layer C: 日内 T+0 子系统（从属于 Layer B）
  ┌─ 开关：由 Layer B 状态决定激活/冻结
  ├─ 额度：min(底仓 × 20%, 可用 Reserve)
  └─ 熔断：日亏 0.3% 或 Layer B 状态变更 → 立即冻结
```

### 3.2 账户级熔断器（Circuit Breaker）

```
高水位线追踪：
  HWM = max(历史所有交易日收盘后的账户总净值)
  每日 15:30 盘后更新 HWM

═══ 盘中软熔断（v2.1 新增）═══

  盘中每 3 秒快照时计算 nav_estimate（持仓市值 + 可用现金）：

  IF nav_estimate ≤ HWM × 0.92:
    → 冻结所有新开仓（试探/加仓/做T买入）
    → 仅允许减仓/止损/卖出操作
    → 告警：「⚠️ 盘中预警：净值逼近熔断线，已冻结新开仓」

  IF nav_estimate ≤ HWM × 0.90:
    → 按 Layer 1 同逻辑执行全部清仓（能卖多少卖多少）
    → 未成交部分区分处理：
       - T+1 锁定份额（当日不可卖）→ 写入 pending_sell_locked，次日 09:30 连续竞价跌停价强平（v1.2 同步 T0 规格书）
       - 跌停/流动性不足导致未成交（份额可卖）→ 写入 pending_sell_unfilled，当日继续重试
    → 告警：「🚨 盘中熔断触发，正在清仓」

  不可撤销规则 — 双层语义（v2.1 R2 二次审计补充）：

    ① 状态不可撤销：
      盘中熔断一旦触发（nav ≤ HWM×0.90），即视为当日进入冷静期。
      即使盘中 V 反转导致收盘 nav > HWM×0.90，以下状态仍不可撤销：
      - 冷静期从当日开始计算，≥ 5 个交易日
      - 盘后硬熔断检查跳过（已由盘中触发覆盖）
      - 不会因 V 反转解除风控状态

    ② 执行可重试：
      清仓执行应持续到收盘，不因“不可撤销”而停止卖出：
      - pending_sell_unfilled 每 30s 检查流动性，有买盘即重新挂卖单
      - 14:55 后升级为 Ask2/Ask3（复用退出执行器跌停策略）
      - 收盘仍未成交 → 转为 pending_sell_locked，次日 09:30 连续竞价跌停价强平（v1.2 同步 T0 规格书）
      理由：V 反转提供了流动性窗口，“不可撤销”不等于“不可重试卖出”。

═══ 盘后硬熔断（原有）═══

触发条件：
  current_nav ≤ HWM × 0.90（从最高净值回撤 10%）

触发动作（不可协商）：
  1. 无视所有子系统信号
  2. 市价清空全部持仓（调用 Layer 1 卖出执行器，共享跌停/pending 全部逻辑）
  3. 停止自动交易，进入 ≥ 5 个交易日强制冷静期
  4. 告警：「🚨 账户熔断：净值从 HWM {hwm} 回撤至 {nav} ({dd:.1%}%)，
           全部清仓，冷静期 5 日」
  5. 冷静期内 ROCm 离线执行参数重优化（Walk-Forward）

解锁条件（全部满足）：
  a) 冷静期 ≥ 5 个交易日
  b) 大盘指数（中证全指 or 万得全A）站上 20日均线
  c) 人工确认解锁（系统发出请求，等待人工 ACK）

状态持久化：
  state.json: {
      "circuit_breaker": {
          "triggered": true,
          "trigger_date": "2026-03-15",
          "trigger_nav": 180000,
          "hwm_at_trigger": 200000,
          "cooldown_expire": "2026-03-22",
          "unlocked": false
      }
  }
```

| 参数 | 值 | 依据 |
|:---|:---:|:---|
| 回撤阈值 | 10% | 20万 × 10% = 2万，可承受但需要暂停反思 |
| 冷静期 | 5 交易日 | 1 周复盘时间 |
| 解锁条件 | 大盘站上 MA20 + 人工 ACK | 防止在系统性下跌中重新进场 |

### 3.3 相关性过滤器（Correlation Filter）

```
计算时机：每日盘后（Phase 1 候选池构建时）
数据：所有备选 ETF 过去 20 日对数收益率

Method:
  corr_matrix = pearson_corr(log_returns, window=20)

  IF Slot A 已持仓 ETF_X，新候选 ETF_Y:
    ρ = corr_matrix[ETF_X][ETF_Y]
    IF ρ ≥ 0.60:
      → 屏蔽 ETF_Y（不进入 Phase 2）
      → 告警：「ℹ️ {ETF_Y} 与持仓 {ETF_X} 相关性 {ρ:.2f} ≥ 0.60，已屏蔽」

  IF 两只候选 ETF_Y, ETF_Z 同日触发信号:
    IF corr(ETF_Y, ETF_Z) ≥ 0.60:
      → 仅保留 Score_entry 更高的一只
      → 告警：「ℹ️ {ETF_Y} 与 {ETF_Z} 高度相关，仅保留 {winner}」
```

> [!NOTE]
> 典型高相关组合（应互斥）：半导体(512480) ↔ 科创50(588000)，
> 光伏(515790) ↔ 新能源(516160)。
> 低相关组合（推荐搭配）：半导体 ↔ 军工，新能源 ↔ 黄金 ETF。

### 3.4 ATR 风险平价仓位计算器（v2.0 新增）

> [!IMPORTANT]
> **Round 3B 结论**：ATR 风险平价的 Sharpe Ratio 比固定 Slot 高约 15-31%（被回测期偏差放大，保守估计 15%+）。
> 核心价值：防止半导体(ATR≈3.6%) 和 红利(ATR≈1.3%) 用同样 7万导致风险暴露差 2.8 倍。
> **实施原则**：CashManager 不需重构（仍锔7万/Slot），仅在 OrderGenerator 层面加计算。

```
公式：
  effective_slot = min(
      risk_budget / (max(ATR_pct, 0.015) * stop_multiplier),
      70000  # 硬上限
  )

其中：
  risk_budget    = current_nav * 0.02（动态跟踪账户净值，下限2500，上限6000）
  ATR_pct        = ATR_20 / Price（归一化波动率，使用信号触发日 T-1 收盘数据）
  stop_multiplier = 3.5（≥ 实际 k=3.0 + 0.5×ATR 滑点/折价余量）
  0.015          = ATR_pct 下限（防止低波缩量期假突破导致超大仓位）

  ⚠️ v2.1 修正说明：
  - stop_multiplier 必须 ≥ 实际止损 k 值。值越大 → 仓位越小 → 越保守。
    原 v2.0 写 2.5 是方向性错误（让仓位比 k=3.0 大 20%），已修正为 3.5。
  - risk_budget 从固定 4000 改为动态 current_nav×2%：
    防止亏损后风险比例被动放大（20万→15万时，2%→2.67% 的越亏越激进问题）。

示例映射（初始 NAV=20万，risk_budget=4000）：
  半导体 (ATR_pct=3.58%) → effective_slot = 4000/(0.0358*3.5) = 31,923
  新能源 (ATR_pct=3.32%) → effective_slot = 4000/(0.0332*3.5) = 34,423
  军工   (ATR_pct=3.12%) → effective_slot = 4000/(0.0312*3.5) = 36,630
  医药   (ATR_pct=2.70%) → effective_slot = 4000/(0.0270*3.5) = 42,328
  消费   (ATR_pct=2.28%) → effective_slot = 4000/(0.0228*3.5) = 50,125
  红利   (ATR_pct=1.25%→强制1.5%) → effective_slot = 4000/(0.015*3.5) = 76,190 → 截断 70,000

与金字塔的联动：
  base_target  = effective_slot * 0.71  # 底仓目标
  scale_1_amt  = effective_slot * 0.19  # 加仓 #1
  scale_2_amt  = effective_slot * 0.10  # 加仓 #2
  trial_amt    = base_target * 0.30    # 试探仓
  confirm_amt  = base_target * 0.70    # 确认仓

CashManager 处理：
  CashManager 继续按 7万/Slot 预留锁定金（不变）
  实际下单量 = effective_slot ≤ 锁定量
  差额自动作为额外缓冲金（接受 Cash Drag 作为风控代价）
```

> [!WARNING]
> **已知局限性（高波踏空）**：当标的处于主升浪顶点（ATR 极高）时，
> 系统只分配极小仓位（如 2.7万）。如果趋势延续，会严重踏空。
> **这是为了保命而必须支付的保险费。** 在波动率极高时轻仓，是规避动量崩塌尾部风险的合理代价。

**T+0**：❌ 冻结（趋势可能转弱，禁止日内博弈）  
**回笼现金**：严格锁定，不再投入日内做T

**跃迁**：
| 目标 | 条件 | 动作 |
|:---|:---|:---|
| → S0 | 退出 Layer 1：price < Stop (k=1.5 保命模式) | 全清，波段终结 |
| → S4 | 回补重建（特殊条件，见第 6 节） | 视为全新建仓周期 |

## 5. 加仓信号定义（Layer B 核心新增）

> [!IMPORTANT]
> 加仓与初始入场是**完全不同的决策**。
> 入场解决"是否进入"（方向判断），加仓解决"是否加码"（趋势延续确认）。
> 加仓信号比入场更严格：**必须已有浮盈 + 回调至支撑 + 抛压衰竭**。

### 5.1 加仓前提条件（全部满足方可进入信号评估）

```
a) 当前持仓处于 S2:BASE 或 S3:SCALED 状态
b) 当前浮盈 ≥ 1.5 × ATR_14（已脱离成本区）
c) 账户级熔断器未触发
d) 退出策略 Score_soft < 0.5（无预警信号，远离减仓线）
e) 距上次加仓 ≥ 3 个交易日（防密集加仓）
f) 加仓后总仓位 ≤ 7万（Slot 上限硬约束）
```

### 5.2 加仓信号定义（四条件共振）

```
必须同时满足以下四项：

1) 日线趋势维持
   KAMA(10) 连续上升 ≥ 2 日  AND  Elder Impulse = Green
   （与入场规格书 S_trend 完全一致的计算方式）

2) 有效回调
   价格从持仓期间最高点回调 ≥ 1.0 × ATR_14
   且回调后未跌破动态跟踪止损线（Chandelier Stop）

3) 筹码支撑位确认
   筹码分布引擎 find_dense_zones() 在回调目标区域
   检测到密集支撑区（density ≥ 筹码集中度前 30%）
   价格触及该支撑区的上沿 ± 0.3 × ATR_14

4) 微观止跌确认（盘中 L1 级别）
   ★ 当日最近 30 分钟内（10 根 3分钟 K 线）：
   a) 成交量 < 前一日同时段成交量 × 0.70（缩量 30%+）
   b) 价格未有效跌破筹码支撑区下沿
   c) 最新 3分钟 K 线的 Close > Open（止跌企稳阳线）
```

### 5.3 加仓执行

```
信号满足后：
  → 在 L1 快照的 Bid1 价位下达限价买单
    （回调买入＝左侧接盘，4-6s 延迟不会造成恶性滑点）
  → 挂单价 = tick_ceil(Bid1)  (买一价，不追)
  → 挂单量 = 加仓目标金额 / 挂单价，取整到 100 份
  → 挂单有效期 = 当日收盘有效（不隔夜）

  成交后：
  → 更新 state.json: position_state → S3 或 S4
  → 重算 T+0 额度上限
  → 重算跟踪止损线（HH 和 ATR 不变，仓位风险敞口扩大）
  → 告警：「✅ {etf_code} 加仓第 {n} 次完成，
           仓位 {old}→{new}万，均价 {avg_cost:.3f}」

  未成交（收盘时）：
  → 自动撤单
  → 不写入 pending（加仓不隔夜等待，次日需重新评估条件）
```

### 5.4 加仓回测方法

```
数据需求：
  日线 OHLCV + 日内 3分钟 K 线 + 筹码引擎日度输出

回测规则：
  1. 筹码支撑位使用 T-1 日盘后计算的密集区
  2. 日线趋势信号使用 T-1 日收盘数据
  3. 盘中缩量确认使用 T 日3分钟K线
  4. 成交价 = 触发时 Bid1 + 0.002 元滑点（限价单保守估计）
  5. 加仓后止损线不上移（仅最高价 HH 更新后自然跟踪）
```

### 5.5 加仓已知局限性

> [!WARNING]
> **踏空主升浪**：当趋势以连续阳线向上逼空（无回调）时，加仓条件永远无法满足。
> 系统将以 71% 仓位（底仓 5万）跑完整个主升浪，错失 29% 仓位的潜在收益。
> **这是为追求低均价和低风险而必须付出的显式代价。**
> 
> **横盘消耗**：长期横盘时筹码峰自然形成，可能触发虚假的"回调至支撑"信号。
> 前提条件 (d) 的 Score_soft < 0.5 会过滤大部分横盘场景。

## 6. 减仓后回补逻辑（S5 → S4 特殊跃迁）

```
前提：当前处于 S5:REDUCED（持仓为原来的 50%）

回补条件（全部满足）：
  a) 减仓后 ≥ 5 个交易日（冷却期）
  b) 价格在减仓后的区间内形成新的筹码密集区
     （横盘整理 ≥ 5 日，筹码引擎确认该区域 density 前 20%）
  c) 新密集区成功经受 ≥ 1 次大盘指数下杀压力测试而未破位
  d) 价格放量突破新密集区上沿
     （vol_ratio ≥ 1.5，与入场 S_volume 同逻辑）
  e) 退出策略 Score_soft = 0（所有预警信号归零）
  f) DeepSeek 情绪评分 > 50（非悲观状态）

回补执行：
  → 视为全新建仓任务！
  → 重新计算当前 ATR，不沿用减仓前的参数
  → 重新计算止损位（基于新的 HH 和 ATR）
  → 买入金额 = 回补至 Slot 目标的 ≤ 100%
  → 跃迁至 S4:FULL
  → T+0 重新激活

不可回补的硬约束：
  → 回补期间 Score_soft 再次 ≥ 0.5 → 立即取消回补计划
  → 绝对不在原价位或更高价位无脑回补（必须有新密集区支撑）
  → 回补只允许 1 次。如果回补后再次触发 Layer 2 → 走 S5→S0 完全退出
```

## 7. Layer C：T+0 与底仓动态联动

### 7.1 核心原则

> [!IMPORTANT]
> **T+0 是底仓的从属子系统，没有独立存在权。**
> 底仓状态变化 → T+0 参数立即同步调整。
> T+0 的风险暴露永远不允许超过底仓的安全边际。

### 7.2 T+0 激活/冻结状态机

```
T+0 状态由 Layer B 驱动，无独立触发能力：

Layer B 状态          T+0 状态      额度上限             理由
─────────────────────────────────────────────────────────────────
S0: IDLE              ❌ OFF        0                    无底仓
S1: TRIAL             ❌ OFF        0                    方向未确认
S2: BASE              ✅ ON*        min(5万×20%, avail)  底仓已确认
S3: SCALED            ✅ ON*        min(6.3万×20%, avail) 仓位扩大
S4: FULL              ✅ ON*        min(7万×20%, avail)   满仓运行
S5: REDUCED           ❌ OFF        0                    趋势可能终结

*ON 条件：current_return > 1% AND 退出 reduced ≠ True AND 日亏 < 0.3%
```

### 7.3 额度动态计算

```
每次 Layer B 状态变化时，立即重算：

Max_T0_Exposure = min(
    current_base_value × 0.20,           # 底仓约束
    cash_manager.available_reserve()      # 现金约束
)

# v2.1 修正（红队审计 P0-★1）：
# 删除原第三项 "slot_target - current_base_value"。
# 理由：正T 买入使用 Reserve 现金，不占 Slot 额度。
# 原第三项在 S4:FULL 时 = 0，导致 T+0 额度为零，与设计意图矛盾。
# available_reserve() 已足够约束可用资金。

示例：
  S2 状态，底仓 5万 → T+0 max = min(1万, reserve) = 1万
  S3 状态，底仓 6.3万 → T+0 max = min(1.26万, reserve) = 1.26万
  S4 状态，底仓 7万 → T+0 max = min(1.4万, reserve) = 1.4万
  S5 状态（减仓后 3.5万）→ T+0 max = 0（§7.2 冻结）
```

### 7.4 Layer B 状态变更时 T+0 的强制同步

```
当 Layer B 发生状态跃迁时：

情况 1: 加仓成功（S2→S3 或 S3→S4）
  → T+0 额度上限自动扩容（下一个快照生效）
  → 已有 T+0 挂单不受影响

情况 2: 退出 Layer 2 减仓 50%（任意→S5）
  → T+0 立即冻结
  → 已有 T+0 未成交挂单 → 立即撤单
  → 告警：「⚠️ Layer 2 减仓触发，T+0 已冻结」

情况 3: 退出 Layer 1 硬止损
  → T+0 立即冻结
  → 已有 T+0 持仓（当日买入的做T仓）：
    IF T+0 持仓方向与 Layer 1 卖出方向相同（有净多头）：
      → 将 T+0 持仓纳入 Layer 1 清仓范围
      → sellable_qty 包含 T+0 的可卖部分
    IF T+0 持仓为当日新买（T+1 锁定）：
      → 写入 pending_sell_locked，次日自动清仓
      → 告警：「⚠️ T+0 持仓 T+1 锁定，次日自动清仓」

情况 4: 信号失效试探仓卖出（S1→S0）
  → T+0 本来就是 OFF，无影响

情况 5: 账户级熔断
  → 所有 ETF 的 T+0 立即冻结
  → 撤销所有 T+0 挂单
  → 全部清仓
```

### 7.5 T+0 与 CashManager 的整合

```
CashManager 优先级定义（从入场规格书继承）：

  LAYER_1_STOP    = 最高（硬止损卖出）
  PREEMPTION      = 高  （抢占买入）
  SCALE_BUY       = 中  （加仓买入）← 新增
  TRIAL_ENTRY     = 中  （试探建仓）
  T0_SCALP        = 最低（日内做T）

当高优先级事件需要资金且 available_cash 不足时：
  → 按优先级从低到高撤销 pending 挂单
  → 与入场规格书 §13.4 CashManager 逻辑完全一致
```

### 7.6 正T/反T 方向选择决策树（v2.0 新增）

> [!IMPORTANT]
> 此决策树解决"做T时应该先买还是先卖"的问题。信号源（KAMA-VWAP 交叉）和具体额度参数留给 Round 4 研究。
> 本节仅定义方向选择的 if-else 逻辑框架和极端行情冻结规则。

```
前提条件（全部满足才进入决策树）：
  a) T+0 处于 ON 状态（由 §7.2 驱动）
  b) 当前时段 NOT IN [09:30-09:45, 14:45-15:00]（禁做T时段）
  c) 全天振幅 = (High - Low) / Open ≥ 1.5%（非死水区）
  d) 无未完结的 T+0 单（上一轮 T+0 已闭环或撤单）

决策逻辑：

  price = L1_Snapshot_Price()
  vwap, sigma = Calculate_VWAP_Bands(period=30min)

  IF price > vwap + 1.5 * sigma:
      # 价格极度向上偏离 VWAP — 高位均值回归概率大 → 考虑【反T】
      IF price_change_today > 6.0%:
          # 极端强势逼近涨停 → 反T飞单风险极高 → 冻结！
          Action = "HOLD"
          Log: 「🔒 T+0 冻结：涨幅 {pct}% 逼近涨停，禁止反T」
      ELIF sellable_qty >= (total_qty * t0_ratio):
          Action = "EXECUTE_REVERSE_T_SELL"
          # 先卖 sellable_qty 的 t0_ratio 部分，挂低位限价买单接回

  ELIF price < vwap - 1.5 * sigma:
      # 价格极度向下偏离 VWAP — 低位均值回归概率大 → 考虑【正T】
      IF price_change_today < -5.0%:
          # 恐慌性砸盘，面临跌停死锁 → 冻结！
          Action = "HOLD"
          Log: 「🔒 T+0 冻结：跌幅 {pct}% 恐慌砸盘，禁止正T」
      ELIF available_cash >= (position_value * t0_ratio):
          Action = "EXECUTE_FORWARD_T_BUY"
          # 用 Reserve 现金买入 t0_ratio 部分，待反弹后卖等量 sellable_qty

  ELSE:
      # VWAP 偏离不足 → 不做T
      Action = "HOLD"
```

> [!NOTE]
> `t0_ratio` 参数（当前为底仓×20%）留给 Round 4 最终确定。
> 此处的 VWAP σ带方向判断框架独立于具体信号——
> 即使 Round 4 改用 KAMA 交叉作为信号，方向选择仍遵循此决策树。

### 7.7 T+0 竞态处理 Mutex（v2.0 新增）

> [!WARNING]
> T+0 与波段止损/止盈同时触发是高概率事件（见顶日往往伴随日内剧烈震荡）。
> 以下三个场景必须有明确的优先级和执行序列。

```
共享互斥锁（Mutex）：T+0 模块与 Layer 1/2 退出模块共享同一把锁。
任何时刻仅允许一个模块提交/修改订单。

────────────────────────────────────────────────────────
场景 1：正T 买单刚成交 → 随后触发 Layer 1 止损

  Timeline:
    14:00:00  正T 买入 1万 成交 → 产生 locked_qty（T+1 不可卖）
    14:00:05  价格急跌，Chandelier Exit 被击穿
    14:00:09  Layer 1 止损信号到达

  处理：
  → Layer 1 享有系统最高优先级（Priority 0）
  → 立即市价清空所有 sellable_qty
  → 正T 产生的 locked_qty（1万）受 T+1 限制，当日无法卖出
  → 写入 pending_sell_locked: {
        "etf_code": "512480.SH",
        "locked_qty": 10000,
        "lock_reason": "forward_t_during_stop",
        "sell_at": "next_day_09:30",  # 次日开盘后连续竞价（v1.2 同步 T0 规格书）
        "sell_strategy": "limit_at_lower_limit"  # 跌停价挂单确保成交
    }
  → 告警：「🚨 正T 仓 {qty} 份 T+1 锁定，次日 09:30 强平」
  → 最大敞口：locked_qty × 10%（跌停） = 1000 元 = 0.5% 总资金

────────────────────────────────────────────────────────
场景 2：反T 刚卖出 → 随后触发 Layer 1 止损

  Timeline:
    13:50:00  反T 卖出 1万（sellable_qty 减少）
    13:50:03  挂低位限价买单等待接回
    13:50:08  价格继续暴跌，触发 Layer 1 止损

  处理：
  → 立即撤销反T 的接回挂单（取消买单）
  → 市价清空剩余 sellable_qty
  → 反T 已卖出的部分 = 客观上完成了"提前逃顶"
  → 属于良性异常（实际减少了亏损暴露）
  → 告警：「ℹ️ 反T 卖出 {qty} 份已完成，接回单已撤销，Layer 1 执行清仓」
  → 无 pending_sell_locked 问题

────────────────────────────────────────────────────────
场景 3：T+0 订单在传输中 → Layer 2 止盈信号到达

  Timeline:
    14:15:00  T+0 买单已提交，处于 SUBMITTED 状态（4-6s 延迟）
    14:15:02  Score_soft 达到 0.9，Layer 2 减仓信号触发

  处理：
  → Layer 2 信号发现存在 SUBMITTED 状态的 T0 订单
  → 进入 WAIT 队列等待（最大等待 = 10 秒）
  → T0 订单状态变化：
      IF T0 成为 FILLED:
        → 该买入量纳入 total_qty
        → 当日新买部分 = locked_qty
        → 重算 sellable_qty = total_qty - locked_qty
        → 执行 Layer 2 减仓 50% 基于 sellable_qty
        → 差额（sellable 不足的部分）写入 pending_sell_locked
      IF T0 超时或 REJECTED:
        → 直接执行 Layer 2 减仓 50% 基于 total_qty
  → 告警：「⚠️ T+0 订单与 Layer 2 竞态，已排队处理」
```

## 15. 编码备忘：红队审计工程项（v2.1 新增）

> [!NOTE]
> 以下4项不影响策略逻辑和胜率，但属于**生产系统必备的工程加固**。
> 编码时需确保覆盖，避免遗漏。无需修改策略参数或交易逻辑。

### 15.1 FSM 故障跃迁路径

```
问题：§4 的 FSM 只定义了 happy-path 跃迁，未覆盖订单异常。

编码时必须处理：
  1. ORDER_REJECTED（停牌/价格笼子/券商风控）
     → 保持原状态不跃迁，记录 scale_attempt_failed，次日重新评估
  2. PARTIAL_FILL（部分成交）
     → 以实际成交量为准更新 qty/avg_cost
     → 成交金额 ≥ 目标 50% → 跃迁至新状态，记录 scale_partial_flag=True
     → 成交金额 < 50% → 保持原状态，已成交部分计入 total_qty
     → 下次加仓目标 = 原目标 - 已成交金额（补足剩余）
  3. TIMEOUT（下单后 10 秒无确认）
     → 查询委托状态：已成交则正常处理，未成交则撤单回退
  4. 重启对账
     → 启动时读取券商实际持仓，与 state.json 对比
     → 不一致时以券商持仓为准，重建 FSM 状态
```

### 15.2 Mutex 超时释放

```
问题：§7.7 的共享 Mutex 若网络卡住会阻塞 Layer 1 止损。

编码时必须确保：
  1. 锁只保护「写状态 + 提交委托」瞬间（< 1 秒）
  2. 等待委托确认必须在锁外进行
  3. 任何持锁超过 2 秒的路径 → 强制释放 + 写 pending 重试
  4. Layer 1 / 熔断 享有锁的抢占权（撤销低优先级持锁操作）
  5. pending 重试安全检查（v2.1 R2 二次审计扩展）：
     重试执行前必须重新验证以下全部条件（AND）：
     a) FSM 当前状态仍为 S2/S3（可能已被止损清仓至 S0）
     b) 全部 6 项加仓前提仍满足
     c) circuit_breaker.triggered == false 且 intraday_freeze == false
        （不得在熔断冷静期或盘中冻结期执行买入）
     d) Slot 有空位（未超 2 只持仓）且相关性过滤通过
     e) 资金可用且不影响止损/熔断卖出优先级
     f) 若任一条件不满足 → 取消 pending，不执行
  6. pending TTL（v2.1 R2 二次审计新增）：
     所有 pending 记录设置 TTL = 1 个交易日
     超时自动作废并告警，避免跨周误执行
```
