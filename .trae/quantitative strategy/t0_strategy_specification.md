## 2. Regime Pre-gate（波动率环境门控）

### 2.1 设计原理

> T+0 的期望收益在低波日为负值。无差别每天做T的策略精确率仅 48.5%，引入 Regime 过滤后提升至 71.5%，年化超额净利润 +22.8%。

### 2.2 复合触发条件

```
T+0 日级使能标志 = regime_active

regime_active = (
    auction_volume_ratio > 1.5          # 集合竞价量比（09:25 确定）
    OR
    atr5_percentile > 65                # 前日 ATR_5 在 60日滚动分位
)

其中：
  auction_volume_ratio = today_auction_volume / MA(auction_volume, 10)
  atr5_percentile = percentile_rank(ATR_5, window=60)
```

> [!NOTE]
> 使用 OR 逻辑而非 AND，理由：  
> - AND 逻辑召回率过低（<45%），会错过大量高波日  
> - OR 逻辑精确率 71.5% + 召回率 68.7% 是更优的权衡  
> - 后续的时间窗口 + 信号阈值会进一步过滤伪信号

### 2.3 计算时点

| 时点 | 动作 |
|:---|:---|
| 09:25 | 集合竞价成交后，计算 `auction_volume_ratio` |
| 09:25 | 读取前日 `ATR_5`，计算 `atr5_percentile` |
| 09:26 | 确定 `regime_active`，写入当日 state |
| **全天不变** | 盘中不重新判定，避免来回翻转 |

### 2.4 参数表

| 参数 | 值 | 来源 |
|:---|:---:|:---|
| 竞价量比阈值 | 1.5 | 国泰君安 (2022) 竞价预测因子研究 |
| ATR 分位数阈值 | 65% | 补充研究 §3A 回测验证 |
| 竞价量比 MA 窗口 | 10 日 | 覆盖 2 周交易日 |
| ATR 分位数窗口 | 60 日 | 约 3 个月滚动 |

## 3. 信号引擎 A：VWAP σ-带买入/卖出

### 3.1 核心机制

价格围绕日内 VWAP 呈尖峰厚尾分布，非正态。在震荡和低波日中，偏离 2σ 后 30 分钟内均值回归概率 ≥ 70%。策略在关键 σ 带位置**预埋限价单**，利用被动 Maker 特性：

- **买入**：VWAP 下方极端偏离处预埋 → 捕获恐慌性超卖
- **卖出**：VWAP 上方极端偏离处预埋 → 捕获脉冲性冲高

### 3.2 VWAP 日内计算（L1 实现）

> [!CAUTION]
> **L1 数据契约（v1.1 P0-5）**：XtQuant L1 快照中的 `volume` 字段为**累计成交量**。  
> VWAP 计算**必须使用增量** `Δvolume = volume - prev_volume`。  
> 校验规则：`Δvolume < 0` 或 `Δamount < 0` → 标记 STALE，跳过该快照。

```python
# 每 3 秒快照更新
delta_vol = snapshot_volume - prev_snapshot_volume   # 增量！非累计
delta_amount = snapshot_amount - prev_snapshot_amount

IF delta_vol <= 0 OR delta_amount <= 0:
    data_quality = STALE   # 异常快照，跳过
    RETURN

cumulative_pv += delta_amount       # 使用增量成交额
cumulative_vol += delta_vol          # 使用增量成交量

vwap = cumulative_pv / cumulative_vol  # 从 09:30 累计

# σ_VWAP 计算：价格与 VWAP 偏差的滚动标准差
deviations.append(snapshot_price - vwap)
raw_sigma = std(deviations[-rolling_window:])  # 滚动窗口

# σ 下限保护（v1.1 P0-3）：防止清淡时段 σ→0 导致无穷大偏离
sigma_floor = price × 0.0005    # 最低 5 bps（≈ 0.5 个 tick @price=1.0）
sigma = max(raw_sigma, sigma_floor)

# 挂单价后处理（v1.1 P0-3）：
# ① tick 对齐：round_to_tick(price, tick_size=0.001)
# ② 涨跌停 clamp：clamp(price, lower_limit, upper_limit)
# 所有挂单价在提交前必须经过此两步处理
```

### 3.3 买入信号定义

```
T0_BUY_SIGNAL 触发条件（全部 AND）：

  ① price ≤ vwap - k_buy × sigma          # 价格低于 VWAP 下方 k_buy σ
  ② regime_active == True                   # Regime 门控通过
  ③ time IN [10:00-11:30, 13:15-14:15]     # 有效时间窗口
  ④ t0_state == IDLE                        # 无未完结的 T+0 交易对
  ⑤ t0_daily_loss < 0.3%                    # 日亏损未触发熔断
  ⑥ Layer_B_state IN [S2, S3, S4]          # FSM 允许
  ⑦ daily_round_trip_count < 1              # 当日尚未完成 RT（v1.1 P1-6）
  ⑧ (k_buy × sigma / price) × 10000 ≥ 25  # 可交易性：预期回归 ≥ 成本（v1.1 P1-6）

触发后动作：
  → 将信号传递给 §7.6 正T/反T 决策树
  → 若决策树返回 FORWARD_BUY → 执行正T买入
  → 预埋限价单价格 = tick_align(clamp(vwap - k_buy × sigma))
```

### 3.4 卖出信号定义

```
T0_SELL_SIGNAL 触发条件（全部 AND）：

  ① price ≥ vwap + k_sell × sigma          # 价格高于 VWAP 上方 k_sell σ
  ② regime_active == True
  ③ time IN [10:00-11:25, 13:15-14:00]     # v1.2：反T卖出截止 14:00（留 15 分钟接回窗口）
  ④ t0_state == IDLE
  ⑤ t0_daily_loss < 0.3%
  ⑥ Layer_B_state IN [S2, S3, S4]
  ⑦ daily_round_trip_count < 1              # 当日尚未完成 RT（v1.1 P1-6）
  ⑧ (k_sell × sigma / price) × 10000 ≥ 25  # 可交易性检查（v1.1 P1-6）

触发后动作：
  → 将信号传递给 §7.6 正T/反T 决策树
  → 若决策树返回 REVERSE_SELL → 执行反T卖出
  → 预埋限价单价格 = tick_align(clamp(vwap + k_sell × sigma))
```

### 3.5 k 值参数（来源：补充研究 §1B 推导链）

| 参数 | 默认值 | 动态调整 | 推导依据 |
|:---|:---:|:---|:---|
| k_buy | **2.0σ** | 上升趋势日 → 1.8σ；下降趋势日 → 2.2σ | 2.0σ 低波日回归概率 79.5%，中位时间 12 分钟 |
| k_sell | **2.8σ** | 高波日 → 2.6σ；低波日 → 3.0σ | 2.8σ 做市商撤单流动性真空点，回落概率 68.5% |
| sigma 滚动窗口 | 60 个快照（3min） | 不动态调整 | 覆盖 1 轮完整的分钟级波动周期 |

> [!WARNING]
> **与 position_management §7.6 的参数统一**：  
> §7.6 正T/反T 决策树使用 1.5σ 作为初始门槛，**本信号引擎的 k_buy/k_sell 必须 ≥ 1.5σ**。  
> 逻辑链：§7.6 的 1.5σ 是"进入决策树"的宽门槛 → 本 §3 的 2.0σ/2.8σ 是"实际下单"的窄阈值。

### 3.6 趋势状态判定（用于 k 值动态调整）

```
日内趋势方向判定：

  IF price > vwap AND kama_10 > kama_10[-60]:  # 价格在VWAP上方且KAMA上行
      trend_state = UPTREND
      k_buy = 1.8σ      # 上升趋势中回调更容易反弹
      k_sell = 2.8σ      # 保持默认

  ELIF price < vwap AND kama_10 < kama_10[-60]:
      trend_state = DOWNTREND
      k_buy = 2.2σ      # 下降趋势中必须要求更深的偏离才接
      k_sell = 2.6σ      # 弱势日冲高更容易回落

  ELSE:
      trend_state = RANGE
      k_buy = 2.0σ      # 默认值
      k_sell = 2.8σ
```

## 4. 信号引擎 B：KDE 筹码支撑位预埋买入

### 4.1 核心机制

利用 ROCm 筹码引擎离线计算的 KDE 密集区，在**盘前**（T-1 日 18:00 更新）确定关键支撑价位，作为候选买入价格写入 `kde_candidate_prices[]`。

> [!WARNING]
> **v1.1 P0-4 修复**：KDE 价位仅**预计算，不直接提交订单**。  
> 原 v1.0 “09:30 开盘即预埋”在逻辑上绕过了 VWAP 必要条件且违反 RED_ZONE 禁令。  
> 实际挂单必须等待 10:00 VWAP gate 通过后才允许提交。

### 4.2 信号定义

```
KDE_BUY_SIGNAL（盘前预计算，盘中等待 VWAP gate）：

  ① dense_zones = chip_engine.find_dense_zones(etf_code, top_n=3)
  ② 选取 最接近当前价下方的密集区上沿 作为候选价位
  ③ 额外过滤：候选价 < vwap_yesterday × 0.98
     → 确保在"有意义的回调"位置而非噪音
  ④ 写入 kde_candidate_prices[]    # 仅個存，不提交订单！

盘中逻辑（v1.1 修复）：
  → 10:00 VWAP gate 通过后，检查 kde_candidate_prices 与
     VWAP σ-带信号的关系：
     → 若 VWAP σ-带信号 落在 KDE 候选价附近（±1 tick）：
        合并为单一买单（避免重复下单）
     → 若不重合：以 VWAP σ-带为主信号，KDE 作为置信度增强
```

### 4.3 与 VWAP 信号的协作规则

| 场景 | VWAP σ-带 | KDE 支撑 | 动作 |
|:---|:---:|:---:|:---|
| 双重确认 | ✅ 触发 | ✅ 价格在密集区内 | 执行买入（最高置信度） |
| 仅 VWAP | ✅ 触发 | ❌ 价位远离密集区 | 执行买入（标准置信度） |
| 仅 KDE | ❌ 偏离不足 | ✅ 价格触碰密集区 | **不执行**（VWAP 为必要条件） |
| 无信号 | ❌ | ❌ | HOLD |

> [!IMPORTANT]
> VWAP σ-带是**必要条件**，KDE 筹码支撑是**充分增强**。单独的 KDE 触发不构成 T+0 买入理由。

## 5. 信号引擎 C：IOPV 溢价率卖出确认

### 5.1 核心机制

ETF 市价相对 IOPV（实时参考净值）的溢价率，反映套利资金的方向性压力。高溢价 = 套利资金即将做空卖出 ETF → 卖出信号增强。

### 5.2 信号定义（卖出辅助）

```
IOPV 溢价率确认（叠加在 VWAP 卖出信号上）：

  premium_rate = (price - iopv) / iopv × 100

  → 当 T0_SELL_SIGNAL 已触发 AND premium_rate > 0.15%:
     → 卖出信号置信度标记为 HIGH
  → 当 T0_SELL_SIGNAL 已触发 AND premium_rate ≤ 0.15%:
     → 卖出信号置信度标记为 NORMAL

  IOPV 确认不改变买卖动作本身，仅影响日志中的置信度标签。
```

> [!NOTE]
> IOPV 不独立触发卖出。在现有框架中（entry_strategy_specification §6.3），  
> IOPV 已用于入场阶段的信号增强，此处复用同一数据管线。  
> **v1.1 增强**：若 IOPV 字段缺失或过时（MISSING/STALE），  
> 置信度固定为 NORMAL，跳过溢价计算，不影响交易动作。

## 6. 交易成本模型 & 利润目标

### 6.1 Round-trip 成本推导

```
单笔 T+0 Round Trip（买+卖）基准金额 V = 14,000 元（v1.1 P1-9 修正）：

  ┌───────────────────────────────────────────┐
  │  佣金（双向）  = max(5, V×0.00015) × 2    │
  │               = 10 元 = 7.1 bps           │
  │                                            │
  │  逆向选择成本  = 1.5 ticks ≈ 15.0 bps     │
  │  （tick_bps = 0.001/1.0×10000 = 10 bps）   │
  │  （注：bps 随 ETF 价格水平变化）             │
  │                                            │
  │  价差捕获收益  = 1.0 tick ≈ -10.0 bps     │
  │  （Maker 被动成交免除 Taker spread cost）   │
  ├───────────────────────────────────────────┤
  │  净 Round-trip 成本 = 7.1 + 15.0 - 10.0  │
  │                     = 12.1 bps             │
  └───────────────────────────────────────────┘

  # v1.1 修正说明：
  # 1. 基准金额从 20k 下调至 14k（匹配 §7.3 额度上限）
  # 2. tick_bps 随价格变化：1 tick = 0.001/price × 10000
  #    price=0.8 → 12.5bps; price=1.0 → 10bps; price=2.0 → 5bps
  # 3. BE 从 11.25 bps 上调至 12.1 bps
  # 4.（v1.2 注释）价差捕获 -10bps 假设回归概率 100%。
  #    若用 79.5% 概率加权：期望价差捕获 = -7.95 bps → BE ≈ 14.15 bps。
  #    25 bps 止盈在两种解释下均有余量（2.07× / 1.77×）。
```

### 6.2 最低止盈目标

```
  理论 BE = 12.1 bps（v1.1 修正）
  安全系数 = 2×（覆盖 KDE 误差 + GUI 异常长尾）
  ─────────────────────────────
  最低止盈目标 = 25 bps = 千分之 2.5

  → 对 ¥14,000 单笔 = 35 元最低利润
  → 对 ETF 价格 1.000 = 至少 0.0025 元价格波动 = 2.5 个 tick
```

> [!CAUTION]
> **硬编码为系统常量**。任何 T+0 round trip 的止盈目标不得低于 25 bps。  
> 如果在 σ 带回归过程中未达到 25 bps → 不提前止盈，等待完整回归至 VWAP ± 0.5σ。

### 6.3 最优单笔金额

| 单笔金额 | Round-trip 佣金率 | 适配性 |
|:---|:---:|:---|
| ¥10,000 | 10.0 bps | ❌ 保底 5 元惩罚严重 |
| **¥14,000** | **7.1 bps** | **✅ 匹配 §7.3 额度上限** |
| ¥20,000 | 5.0 bps | ✅ 理论更优（但超 §7.3 上限） |

> [!NOTE]
> 由于 position_management §7.3 的 T+0 额度上限 = 底仓 × 20% ≤ 1.4万，  
> 单笔金额实际约束在 **1.0万 ~ 1.4万** 范围。  
> **结论**：在现有额度框架下，成本可控但不宽裕。必须严格执行 25 bps 止盈底线。

## 7. 订单管理协议（预埋限价单）

### 7.1 核心范式：Maker 预埋而非 Taker 追价

```
基本原则：
  ✅ 预先挂单（深排队 Maker） → 被动等待成交
  ❌ 信号触发后市价追单（Taker） → 4-6s 延迟 = 追到更差价格

策略分时逻辑：
  09:26  Regime Pre-gate 判定
  09:30  KDE 候选价位就绪（仅预计算，不提交订单）     # v1.1 修复
  10:00  VWAP σ-带引擎激活 → 动态计算预埋价位
         KDE 候选价与 VWAP 信号进行匹配，符合条件时提交挂单   # v1.1 修复
         每 3 分钟（60 个快照）重新评估 σ 值
         仅当新计算的预埋价位 与当前挂单 偏离 > 2 tick 时才修改挂单
  14:00  反T卖出信号截止（v1.2），此后仅允许接回买入          # v1.3 补注
  14:15  清道夫：撤销所有未成交买入预埋单（含反T接回单）
  14:55  全部 T+0 操作结束
```

> [!CAUTION]
> **操作禁令条款（v1.4 新增，终审 6.2）**  
> T+0 子系统运行期间，操作者 **严禁** 执行以下行为：  
> 1. 手动修改 T+0 挂单的价格或数量  
> 2. 手动撤销或新增 T+0 挂单  
> 3. 手动调整 k_buy / k_sell 等运行期参数  
> 4. 在系统自动挂单等待期间，因「总想做点什么」而主动干预  
>  
> **理由**：手动修改会破坏 Maker 预埋策略的概率边际。例如将挂单价从 VWAP-2σ 改为 VWAP-1.5σ，  
> 成交概率增加但赢率下降，导致未经验证的参数变更破坏原有的概率优势。  
> **唱一允许的人工操作**：在告警提示人工介入时（§7.5 对账失败）响应系统请求。

### 7.2 预埋单成交概率基准

| 预埋位置 | 日内被触及概率 | 触及后完全成交概率 | 排队等待时间中位数 |
|:---|:---:|:---:|:---:|
| Bid1/Ask1 | 100% | 85-90% | 45秒 - 1分钟 |
| VWAP ± 1.5σ | 15-25% | 45-55% | 不定 |
| VWAP ± 2.0σ | 5-8% | 35-40% | 不定 |

> **系统必须坦然接受 2.0σ 预埋单成交概率仅 ~40%**。大多数交易日 T+0 不会执行任何交易。这是正确的——宁可不做也不滥做。

### 7.3 部分成交处理协议

```
IF partial_fill_detected (成交量 < 目标量的 80%):
    → 启动 3 分钟容忍倒计时
    → 3 分钟内完全成交 → 正常处理
    → 3 分钟后仍部分成交：
```

> **v1.3 P0 增强：超时后必须强制对账，不能仅冻结**（第三轮红队 2.1）

### 9.2 日内亏损熔断（继承 §7.2）

```
每笔 T+0 闭环后更新日内累计损益：

  t0_daily_pnl += round_trip_pnl  # 含佣金

  IF t0_daily_pnl <= -(total_nav × 0.003):   # 0.3% = 约 600 元（20万）
      → T+0 当日冻结，不可恢复
      → 撤销所有 T+0 挂单
      → 告警：「🚨 T+0 日亏损触及 0.3%，当日冻结」
      → 该日 t0_frozen_until = "next_trading_day"
```

### 9.2a T+0 子系统周累计亏损熔断（v1.1 P1-12）

```
滚动 5 个交易日的 T+0 累计损益：

  t0_weekly_pnl = sum(t0_daily_pnl[-5:])

  IF t0_weekly_pnl <= -(total_nav × 0.005):   # 0.5% = 约 1000 元（20万）
      → T+0 子系统冻结至下一周一
      → 告警：「🚨 T+0 近 5 日累计亏损触及 0.5%，冻结至下周一」
      → 防止慢性失血（每天亏50-100元但不触发0.3%日熔断）
```

### 9.2b T+0 子系统月度滚动亏损熔断（v1.2 新增）

```
滚动 30 个交易日的 T+0 累计损益：

  t0_monthly_pnl = sum(t0_daily_pnl[-30:])

  IF t0_monthly_pnl <= -(total_nav × 0.01):   # 1.0% = 约 2000 元（20万）
      → T+0 子系统冻结 30 个交易日
      → 告警：「🚨 T+0 近 30 日累计亏损触及 1.0%，冻结 30 天并参数复盘」
      → 防止“每天小亏但不触发日/周熔断”的慢性失血
      → 红队审计 6.5 发现：每日亏 15bps×1.4万=21元，仅周熔断无法捕获
```

### 9.2c T+0 交易级连续亏损熔断（v1.3 新增）

> [!IMPORTANT]
> **第三轮红队测试区 4.1 发现**：日/周/月三层熔断均基于累计金额，无法捕获「每笔亏损15bps，
> 连续3笔亏损但总额未触线」的模式。需要一个基于**笔数**而非金额的短路保护。

```
连续亏损笔数计数器（v1.3 新增）：

  每笔 T+0 闭环后判定：

    IF round_trip_pnl < 0:
        consecutive_loss_count += 1
    ELSE:
        consecutive_loss_count = 0    # 盈利即重置

    IF consecutive_loss_count >= 3:
        → T+0 冻结至下一交易日
        → 告警：「🚨 T+0 连续 3 笔亏损，冻结至明日（信号可能失效）」
        → 冻结解除后 consecutive_loss_count 重置为 0

  设计理由：
    - 连续亏损 3 笔 = 均值回归假设可能在当日环境下失效
    - 此熔断比日亏损熔断更灵敏：3 × 15bps × 1.4万 = 63元 < 0.3% 日线
    - 能在累计金额达到日线（600元）之前就短路保护
    - 盈利一笔即重置，不惩罚正常的偶发亏损
```

```
  IF price_change_today > 6.0%:   → 禁止反T（涨停卖飞风险）
  IF price_change_today < -5.0%:  → 禁止正T（跌停锁仓风险）
```

### 9.4 T+0 Mutex 竞态处理（继承 §7.7 + v1.1 新增场景 4）

完整竞态处理见 [position_management §7.7](file:///d:/新建文件夹/position_management_specification.md)。

**四个**核心场景的处理优先级：
- 场景 1（正T成交 → Layer 1 止损）：止损优先，正T locked_qty 次日强平
- 场景 2（反T卖出挂单未成交 → Layer 1 止损）：撤接回单，良性异常
- 场景 3（T+0 传输中 → Layer 2 止盈）：等待 10s 后执行
- **场景 4（v1.1 P0-2 新增）：反T 接回已成交 → Layer 1 止损**

```
场景 4：反T 接回买单已成交 → Layer 1 止损触发（v1.1 新增）

  Timeline:
    13:30:00  反T 卖出 1万 成交
    13:40:00  接回买单 1万 @低价 成交 → 产生 locked_qty = 1万
    13:45:00  价格急跌，Layer 1 止损触发

  处理：
  → 与场景 1 相同：Layer 1 最高优先级
  → 立即市价清空所有 sellable_qty
  → 接回产生的 locked_qty = 1万，T+1 锁定，当日无法卖出
  → 写入 pending_sell_locked（同场景 1 处理流程）
  → 告警：「🚨 反T 接回仓 {qty} 份 T+1 锁定，次日 09:30 强平」

  与场景 1 的关键区别：
  场景 1 的 locked_qty 来自正T 买入（新的多头暂露）
  场景 4 的 locked_qty 来自反T 接回（恢复性买入，同样形成新的多头暂露）
  处理逻辑完全相同，仅 lock_reason 不同："reverse_t_buyback_during_stop"

  最坏敗口量化（v1.2 增补）：
    反T 卖出 @1.050 → 接回 @1.035 → T+0 利润 = 150 元
    次日强平 locked_qty @跌停价(0.9315) → 亏损 = 1,035 元
    NET = 150 - 1,035 = -885 元 = 0.44% 总资金
    ETF 连续跌停概率 <0.1%，风险极低但已知不可调和。
```

> [!NOTE]
> **locked_qty 次日处理时间统一（v1.1 P1-11）**：  
> 所有 pending_sell_locked 的强平挂单统一在 **09:30 开盘后立即执行**（而非 09:25 集合竞价）。  
> 理由：集合竞价涢价不可控，连续竞价可以用限价单追求更好价格。

```
       卖出 1.2万（从 sellable_qty 中扣除）
```

```
目标：验证完整执行链路 + 收集真实市场微观数据

  - XtQuant 模拟账户（miniQMT 或 QMT 模拟环境）

  - 与实盘完全相同的代码路径（无 if-simulation 分支）

  ① regime_active 判定：竞价量比 + ATR 分位 → 精确率实测

  ③ 每笔 RT 的毛利 bps / 净利 bps / 滑点 bps
  ④ easytrader 端到端延迟分布（P50/P95/P99）
  ⑤ GUI 操作次数 / 日内冻结次数
  ⑥ 四层熔断触发次数

通过标准（全部满足方可进入阶段 2）：
  a) 信号触发 ≥ 3 次（v1.4 修订，原 10 次不现实）

  d) 无幽灵成交事件（§7.5 对账全部 CASE 通过）

  v1.4 修订说明：

  - 触发标准从 10 次降至 3 次：60 天 ACTIVE 30 天 × 8% = 2.4 次，
```
