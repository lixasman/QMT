# 交易系统流程架构图（分层：筛选条件 / 触发条件 / 降级条件）

> 目的：把本仓库的“**T-1 盘后准备 → T 日盘中交易 → T 日盘后扫描**”完整链路讲清楚。  
> 阅读入口：`main.py` → `strategy_runner.py`（策略主循环），其余模块围绕它提供外部因子、入场/出场/持仓/T0 子策略与适配器。

---

## 0. 一句话总览（你要建立的心智模型）

1. **T-1 盘后**：产出“给交易系统消费”的外部因子快照（筹码/微观 + 情绪）到 `output/integration/`  
2. **T 日盘前**：构建当日候选池（`watchlist`），必要时自动补齐 T-1 因子  
3. **T 日盘中**：按 `tick_interval_s` 进行逐笔循环（tick），对 **待确认入场**（`pending_entries`）做“阶段3（Phase3）确认”并下单；对 **持仓** 执行“退出（Exit）Layer1/Layer2”、救生衣回补、日内做T（T0）、加仓（Scale）、组合熔断  
4. **T 日盘后**：对候选池做“阶段2（Phase2）扫描”，生成下一交易日的 `pending_entries`

> 关键约束：交易系统读取外部因子时，**只取“日期严格 < 当天”的最新文件**，保证盘中不会误用当日未完成/不一致的数据（见 `integrations/watchlist_loader.py` 的“pick_latest_before”逻辑）。

## 0.1 术语与缩写对照（尽量中文化）

- 候选池：`watchlist`
- 待确认入场列表：`pending_entries`
- 外部因子快照：`output/integration/` 下的 chip/sentiment 文件
- “阶段2/阶段3”：Phase2（盘后扫描）/ Phase3（盘中确认）
- “逐笔循环”：tick（按 `tick_interval_s` 周期执行）
- 关键缩写：
  - ATR：平均真实波幅（用于跳空保护、回撤与止损尺度）
  - VWAP：成交量加权均价（用于入场确认与 T0 信号）
  - IOPV：ETF 实时净值估计（用于入场溢价门控）
  - PR：`profit_ratio`（盈利盘比例/筹码盈利比例）
  - NAV：净值（`nav`）
  - HWM：历史净值高水位（`hwm`）
  - KDE：核密度估计（用于 T0 支撑区合并，可缺失降级）
  - DPC：`dpc_peak_density`（筹码峰值密度，用于退出侧 `S_chip`）

## 0.2 关键评分 / 信号名词解释（避免代词）

- `score`（Phase2 入场评分）：`entry/scoring.py` 计算，五信号加权
  - 权重：`S_squeeze 0.30`、`S_volume 0.25`、`S_chip_pr 0.20`、`S_trend 0.15`、`S_micro 0.10`
  - 取值范围：`0.0 ~ 1.0`
  - 触发：`score >= 0.45` 且 `diversity_gate` 成立（`S_volume`/`S_chip_pr`/`S_trend` 至少一个 > 0）
  - 强信号：`score >= 0.70`
- `score_soft`（Exit 退出软评分）：`exit/scoring.py` 在盘中每个 tick 计算，供 Layer1 / Layer2 / 救生衣使用
  - 公式：`0.7*S_chip + 0.7*S_sentiment + 0.5*S_diverge + 0.4*S_time`（四项均为 0~1）
  - 取值范围：`0.0 ~ 2.3`（保留 2 位小数）
  - 降级：任一信号 `DataQuality.UNAVAILABLE` 时，该信号按 `0.5` 参与（中性风险假设）
- `score_soft_layer1 / score_soft_layer2`：当前实现相同（`strategy_runner.py` 直接把 Layer2 结果赋给 Layer1）
- `S_chip`：筹码风险信号，基于 `dpc_peak_density` 近 5 日趋势 + `profit_ratio >= 75` 的门槛；输出离散值 `{0, 0.3, 0.5, 0.7, 1.0}`
- `S_sentiment`：情绪风险信号，`sentiment_score_01 <= 0.35` 记为 1，否则 0（情绪越弱，退出风险越高）
- `S_diverge`：背离风险信号，满足 RSI 顶背离且（ADX 转弱 或 量缩新高）记为 1，否则 0
- `S_time`：时间/回撤风险信号，`days_held > 15` 且 `days_since_high >= 6` 且 `return >= 5%` 记为 1，否则 0
- `DataQuality`：数据质量枚举；`OK` 正常、`STALE` 行情过旧、`UNAVAILABLE` 信号缺失（用于 score_soft 降级与 Layer1 分支）
- `sentiment_score_01`（情绪分 0~1）：来自 `output/integration/finintel/sentiment_<code6>_YYYYMMDD.json`
  - 若缺失：默认 `0.5`
  - 仅用于**退出侧**：参与 `S_sentiment` → `score_soft`（不参与 Phase2 入场评分）
  - 规则：`<= 0.35` 记为 1（风险），否则 0
- `sentiment_score_100`（情绪分 0~100）：同上 JSON 的 100 分制版本
  - 若缺失：默认 `50`
  - 仅用于**入场候选池过滤**：`sentiment_score >= 60`（`watch_auto_no_filter` 可跳过）
  - 不参与 Phase2 入场评分，仅决定候选池是否保留

**情绪评级 → 分值对照表（FinIntel 评级映射）**

| 评级 | sentiment_score_01 | sentiment_score_100 | S_sentiment（退出侧） | 入场候选池情绪门槛（>=60） |
|---|---:|---:|---:|---:|
| A | 0.90 | 85 | 0 | 通过 |
| B | 0.65 | 70 | 0 | 通过 |
| C | 0.50 | 50 | 0 | 不通过 |
| D | 0.25 | 30 | 1 | 不通过 |
| E | 0.10 | 15 | 1 | 不通过 |

> 说明：`S_sentiment` 规则为 `sentiment_score_01 <= 0.35` 记为 1，否则 0；`sentiment_score_100` 仅用于候选池过滤。

## 1. 两日工作流（T-1 → T）

### 1.1 T-1 盘后（建议 15:30 后）

**(A) 计算筹码/微观因子（交易系统消费的唯一路径）**

```bash
python -m etf_chip_engine.daily_batch --date auto
```

- 关键输出（交易系统消费）：
  - `output/integration/chip/batch_results_YYYYMMDD.csv`
- 重要筛选（流动性预筛，默认开启）：
  - 回看窗口（lookback）=60 天、活跃天数≥45、成交额中位数≥2,000,000
  - 通过稳定计算池（stable_pool）复用（默认 30 天刷新一次），减少每日池子漂移
- `--date auto` 的日期语义在 README 中有说明：交易日 15:30 前用 T-1，15:30 后用 T；非交易日用最近交易日（见 `README.md` 的 “ETF Daily Batch Download And Date Semantics” 段落）

**(B) 生成情绪因子（交易系统消费的是 integration 目录里的 JSON）**

```bash
# 热门池（产生“热门列表 CSV”；用于 watch_auto 自动候选池）
python -m finintel --signal-hot-top 15 --no-trace

# 单只 ETF（产生 integration/finintel/sentiment_* 文件）
python -m finintel --signal-etf 159107 --no-trace
```

- 关键输出（交易系统消费）：
  - `output/integration/finintel/sentiment_<code6>_YYYYMMDD.json`
  - `output/finintel_signal_hot_YYYYMMDD.csv`（仅用于热门池/自动候选池）

### 1.2 T 日盘中（开盘前启动即可）

典型启动（示例来自 README，参数以你的环境为准）：

```bash
python main.py --adapter xt --xt-path "<miniQMT_userdata_path>" --xt-account "<account_id>" --xt-session 1 ^
  --position-sizing-cash 200000 ^
  --state-path "data/state/portfolio_sim_<account_id>.json" ^
  --phase2-no-reentry-after-confirm ^
  --phase2-skip-high-chase-after-first-signal ^
  --phase2-high-chase-signal-source all_signals ^
  --phase2-high-chase-lookback-days 60 ^
  --phase2-high-chase-max-rise 0.15 ^
  --exit-layer2-threshold 0.7 ^
  --exit-atr-pct-min 0.025 ^
  --exit-atr-pct-max 0.04 ^
  --exit-k-accel ^
  --exit-k-accel-step-pct 0.05 ^
  --exit-k-accel-step-k 0.2 ^
  --exit-k-accel-k-min 1.0 ^
  --auto-prep --watch-auto --watch-auto-no-filter --hot-top 10
```

- `--position-sizing-cash`：按账户资金自动线性推导仓位上限与风险预算；例如 `200000 -> slot=35000, risk_min=1250, risk_max=3000`
- `--state-path`：指定策略本地状态账本文件；模拟盘/实盘/不同账户强烈建议分开，避免把旧状态残留带入本次运行
- `--auto-prep`：盘前自动检查并补齐 T-1 因子（缺就补，失败就告警并继续）
- `--watch-auto`：候选池基于“默认 50 只 ETF 基池 + 静态 `--watch`”，再并入 `output/finintel_signal_hot_<T-1>.csv` 的热门代码
- `--watch-auto-no-filter`：不做阈值过滤（否则会按 `entry/watchlist.filter_watchlist` 过滤）
- `--phase2-no-reentry-after-confirm`：确认单成交后，同一 ETF 在持仓未清掉前不再重复生成新的入场单
- `--phase2-skip-high-chase-after-first-signal` + `all_signals/60天/15%`：若当前信号相对窗口内首信号已上涨超过 15%，则阻断追买

## 3. 分层细化（每层：筛选条件 / 触发条件 / 降级条件）

### 第0层：新闻抓取（`newsget/`）

- **筛选条件**
  - 财联社热度 Top5 + 东方财富热度 Top5（合计 Top10）
- **触发条件**
  - 手动：`python -m newsget`
  - 或作为 FinIntel 管线的输入：`finintel/pipeline.py` 会通过 `newsget/ingestion.py` 拉取
- **降级条件**
  - 单一来源抓取失败：仅输出告警，仍返回另一来源新闻
  - 正文抓取失败：该条 `content=None`，不阻断整体
- **关键实现**
  - `newsget/ingestion.py`

### 第1层：情绪/情报因子（`finintel/`）

> 交易系统主要消费：`output/integration/finintel/sentiment_<code6>_<YYYYMMDD>.json`

- **筛选条件**
  - 热门池：`--signal-hot-top N` 选热门前 N（内部还支持“主题分散 + 每主题最多 M 只”的再筛选）
  - 可强制把“你关心的持仓/关注标的”注入热门池，避免被热门筛选丢掉（见 `finintel/main.py` 的持仓注入逻辑）
- **触发条件**
  - 手动运行 FinIntel
  - 或由盘前 `--auto-prep` 调用（见 `integrations/premarket_prep.py`）
- **降级条件**
  - `--hot-fast`：热门池快速模式，跳过新闻/持仓/资金流抓取（信息降级但更快）
  - `sentiment_struct` 不完整 / 评级字段非法（grade）：从输出文本里抽取“综合评级”并映射为 A~E，置信度降级为低（LOW）
  - `sentiment_score_01 / sentiment_score_100` 解析失败：按映射表回退
- **关键实现**
  - `finintel/main.py`（写入 integration JSON 的函数 `_write_signal_integration_json`）

### 第2层：筹码/微观因子（`etf_chip_engine/` 日批量）

> 交易系统主要消费：`output/integration/chip/batch_results_<YYYYMMDD>.csv`

- **筛选条件（预筛池）**
  - 流动性预筛默认开启（配置在 `etf_chip_engine/config.py`）：
    - `liquidity_prefilter_lookback_days=60`
    - `liquidity_prefilter_min_active_days=45`
    - `liquidity_prefilter_min_median_amount=2,000,000`
  - 稳定计算池（stable_pool）默认开启：复用一份稳定计算池，默认 30 天刷新一次
- **触发条件**
  - 手动：`python -m etf_chip_engine.daily_batch --date auto`
  - 或盘前 `--auto-prep` 检测到缺失后调用补齐（交易日参数 `trade_date=T-1`）
- **降级条件**
  - 稳定计算池（stable_pool）状态不可复用：自动重建稳定池；若保存状态失败，仅告警继续
  - 不满足流动性预筛：该标的直接跳过（不下载逐笔数据 tick、不计算）
- **关键实现**
  - 阈值：`etf_chip_engine/config.py`
  - 稳定计算池（stable_pool）/预筛行为：`etf_chip_engine/service.py`

### 第3层：盘前自动补齐（`--auto-prep`，`integrations/premarket_prep.py`）

- **筛选条件**
  - 仅当目标文件不存在时才补齐（避免重复跑耗时任务）
- **触发条件**
  - `main.py` 启动参数 `--auto-prep`（对应 `StrategyConfig.auto_prep=True`）
- **降级条件**
  - 无法解析 T-1 交易日：直接跳过补齐（告警）
  - 补齐 chip 失败 / 补齐 finintel 失败：告警并继续交易主循环（不静默）
- **做了什么**
  1. 若 `output/integration/chip/batch_results_<T-1>.csv` 缺失 → 调 `etf_chip_engine.daily_batch.run_daily_batch(trade_date=T-1)`
  2. 若 `output/finintel_signal_hot_<T-1>.csv` 缺失（且 `hot_top>0`）→ 调 `finintel --signal-hot-top hot_top`
  3. 对 “静态 watch + 当前持仓 + 热门池代码” 的并集，逐个确保存在 `output/integration/finintel/sentiment_<code6>_<T-1>.json`，缺则调 `finintel --signal-etf <code6>`

### 第4层：候选池构建（watch_auto + 外部因子装配 + 阈值过滤）

#### 4.1 候选代码集合来源（watch_auto）

- **筛选条件**
  - 若启用 `--watch-auto`：
    - 先取默认 50 只 ETF 基池（`backtest.universe.DEFAULT_UNIVERSE_CODES`）
    - 再与静态 `--watch` 合并去重
    - 若存在 `output/finintel_signal_hot_<T-1>.csv`，再把其中 code 列并入
- **触发条件**
  - 盘前 `pre_open`：用于生成当日监控池
  - 盘后 `post_close`：用于 Phase2 扫描
- **降级条件**
  - 热门列表 CSV（hot csv）缺失 / 解析失败：
    - 默认：降级回退到“50 只 ETF 基池 + 静态 `--watch`”
    - 若（配置项）要求必须有热门列表 CSV：则直接返回空候选池（当前 CLI 未暴露该开关，但代码有预留字段）
- **关键实现**
  - `strategy_runner.py` 的 `_resolve_watch_codes`

#### 4.2 外部因子装配（chip + sentiment → WatchlistItem）

- **筛选条件**
  - 外部因子文件选择规则：只取“日期严格 < 当天”的最新文件
    - chip：`output/integration/chip/batch_results_*.csv`
    - sentiment：`output/integration/finintel/sentiment_<code6>_*.json`
- **触发条件**
  - `_build_watchlist()` 调用时触发加载
- **降级条件**
  - chip CSV 缺失：所有标的筹码/微观字段降级为缺失（profit_ratio=0，微观字段 None）
  - chip CSV 存在但找不到该 ETF 行：该 ETF 筹码/微观字段降级为缺失
  - sentiment JSON 缺失：`sentiment_score_100=50`，`sentiment_score_01=0.5`
  - 任意异常：整体降级为“默认因子”（每只 ETF `sentiment_score=50`、`profit_ratio=0`）
- **关键实现**
  - `integrations/watchlist_loader.py`

#### 4.3 阈值过滤（仅当启用 `watch_auto` 且未指定 no_filter）

- **筛选条件（硬阈值）**
  - `sentiment_score >= 60`
  - `profit_ratio >= 75`
- **软标记（不拦截，作为风险提示字段）**
  - 若 `vpin_rank > 0.70` 或 `ofi_daily < 0` → `micro_caution=True`
- **触发条件**
  - `--watch-auto` 且未启用 `--watch-auto-no-filter`
- **关键实现**
  - `entry/watchlist.py` 的 `filter_watchlist`

### 第5层：盘中入场确认（Entry Phase3，处理 `pending_entries`）

> 盘后 Phase2 产生 `PendingEntry`（状态 `PENDING_TRIAL`），盘中 Phase3 负责确认并下单。

- **触发条件**
  - 每个逐笔周期（tick）：`strategy_runner.py` 会遍历 `state.pending_entries`，处理状态为 `PENDING_TRIAL` / `PENDING_CONFIRM` 的项
- **筛选条件（确认前置条件）**
  - `desired_qty > 0`
  - 未过期：`now.strftime("%Y%m%d") <= expire_yyyymmdd`
  - 时间窗：`now.time() <= 14:30`（`ENTRY_CUTOFF_TIME`）
  - 行情数据质量门控：L1 不可为 STALE/MISSING/UNAVAILABLE（被 `core.validators.assert_action_allowed` 阻断）
- **确认条件（全部通过才下单）**
  1. **价格突破**：`last_price > H_signal`
  2. **跳空保护**：`gap_ratio <= max(1%, 0.5*ATR20/Close_T)`
  3. **VWAP 斜率**：09:50 前处于热身期，跳过斜率；之后要求斜率为正
  4. **IOPV 溢价**：试仓 ≤0.5%，确认 ≤0.3%；若 iopv 缺失则该条件直接放行（降级跳过）
- **降级条件**
  - iopv 缺失：跳过溢价检查
  - VWAP 热身期：跳过斜率检查（用于避免早盘噪声）
  - 成交回报字段无法解析：用“回退填充”（并输出显式降级告警）
  - `confirm_order` 返回未知（UNKNOWN）：不判失败，下个逐笔周期（tick）继续重试确认
- **关键实现**
  - 规则：`entry/phase3_confirmer.py`
  - 下单与锁资：`entry/entry_fsm.py` 的 `apply_confirm_action`
  - 调用点：`strategy_runner.py` 的 `_process_pending_entry` / `_process_placed_orders`

### 第6层：盘中退出与风控（Exit Layer1/Layer2 + 救生衣回补）

#### 6.1 Layer1（硬触发：止损/死水/跳空保护）

- **触发条件（三选一）**
  1. **破止损**：`last_price < stop_price`
  2. **死水区**：`days_held >= 10` 且 `|return| <= 2%`
  3. **跳空保护**：仅在 `09:25` 与 `13:00` 这两个“分钟”触发点检查，且 `last_price < stop_price * 0.97`
     - 说明：此检查按“分钟粒度”严格匹配；若主循环只在 09:30 后进入盘中，则 09:25 这一触发点可能不会被执行
- **筛选条件**
  - L1 行情数据必须 `DataQuality.OK`，否则 Layer1 整体跳过（显式降级告警）
- **决策分支（仅对“破止损”路径）**
  - 若存在任何 `DataQuality.UNAVAILABLE`（预留扩展）→ 最高风控：直接全清（不使用救生衣）
  - 若 `score_soft > 0` → 直接全清（退出软评分为正，视为风险确认）
  - 若 `score_soft == 0` 且未用过救生衣 → 执行“救生衣 70/30”：卖出 70%，保留 30%
  - 若 `score_soft == 0` 且已用过救生衣 → 直接全清
- **降级条件**
  - Chandelier 止损线计算失败：回退为参考价的 97%（`avg_cost` 优先，其次 `last_price`）

#### 6.2 Layer2（软触发：评分≥阈值 → 减仓 50%）

- **触发条件**
  - 计算 `score_soft`，若 `score_soft >= 0.7` → 减仓 50%（整手取整），并把仓位状态置为 `S5_REDUCED`
- **筛选条件**
  - 仅在可卖数量 `sellable_qty>0` 时才会下单
- **降级条件（逐信号降级，确保可运行）**
  - 日线 bars 拉取失败：依赖 bars 的 `S_diverge` / `S_time` 标记为 `UNAVAILABLE`（评分按 0.5 参与）
  - `S_diverge / S_time / S_sentiment` 任一计算异常：该信号标记 `UNAVAILABLE`（评分按 0.5 参与）
  - `S_chip`：
    - 冷启动/不可用（`chip_days < 10` 或 dpc 不足 5 点）：标记 `UNAVAILABLE`（评分按 0.5 参与）
    - 计算异常：标记 `UNAVAILABLE`（评分按 0.5 参与）
  - 若发生“灾难性失败”（导入/总评分函数异常）：直接使用全降级分数 1.15

#### 6.3 救生衣回补（Lifeboat Buyback）

- **触发条件（6 条件全部满足）**
  1. 冷却期：自卖出时刻起，**交易分钟**累计 ≥ 30（午休不计入）
  2. 价格回到止损线上方：`last_price > stop_price`
  3. Layer2 评分为 0：`score_soft == 0.0`
  4. 行情不为 STALE（且 ExitFSM 额外要求必须为 OK）
  5. 非“跌停开板死猫跳”：`last_price > limit_down * 1.02`
  6. 时间不晚于 14:30（回补截止）
- **回补数量**
  - 估算“原仓位”为当前仓位的 10/3，然后买回差额，并按 100 股整手取整；取整后 ≤0 则拒绝并记录原因
- **降级条件**
  - 条件不满足：写拒绝日志，不下单
  - 整手取整导致数量≤0：按 LOT_ROUNDING 拒绝并记录

- **关键实现**
  - `exit/layer1.py`、`exit/layer2.py`、`exit/lifeboat.py`、`exit/exit_fsm.py`
  - 主循环调用点：`strategy_runner.py` 的 `_process_position_tick` 与 `_run_opening_gap_checks`

### 第7层：T0（日内做T）

- **触发条件（启用门槛）**
  - 仓位状态必须在 `{S2_BASE, S3_SCALED, S4_FULL}`（减仓态 `S5_REDUCED` 禁用）
  - 非冻结：`t0_frozen=False`
  - 浮盈门槛：`return > 1%`
  - 日内 T0 亏损限制：`daily_t0_loss < 0.3%`
  - 禁止时间窗：`09:30–09:45`、`14:45–15:00`（直接返回 HOLD）
- **信号触发（VWAP 带）**
  - 早于 10:00 不触发
  - 价格触及 `vwap ± k*sigma`，并满足时间窗允许买/卖
  - 额度：单笔金额 10,000~14,000（且可用资金允许）
  - 可交易性：`tradability_bps >= 25`
- **降级条件**
  - KDE 支撑区加载失败：不使用 KDE 合并（仍可做T）
  - 触发 breaker（按日/5日/30日/连续亏损）：当日冻结 T0
  - 下单失败 / 锁资失败 / 订单对账异常：进入冻结模式并停止 T0
- **关键实现**
  - 启用门槛：`position/t0_controller.py`
  - 信号引擎：`t0/signal_engine.py` + `t0/time_window.py` + `t0/breaker.py`
  - 执行入口：`position/position_fsm.py` 的 `execute_t0_live`

### 第8层：加仓（Scale）

- **前提筛选（全部满足才评估信号）**
  - 仅允许在 `S2_BASE / S3_SCALED`
  - 盈利达到阈值：`unrealized_profit_atr14_multiple >= 1.5`
  - 组合熔断未触发，且盘中未冻结
  - 退出评分较低：`score_soft < 0.5`
  - 加仓最小间隔：≥3 天
  - 不超过 `effective_slot`（仓位上限）
- **信号触发（全部满足才执行加仓）**
  1. 趋势：KAMA 上升天数≥2 且 ElderImpulse 为绿
  2. 回撤：回撤≥1 ATR 且仍在止损线上方
  3. 筹码支撑：密集区密度处于前 30%（rank≥0.70）且“距最近密集区”≤0.30 ATR
  4. 微观确认：波动收缩比≤0.70，且支撑未破，且收盘强势
- **降级条件**
  - `dense_zones_json` 解析失败/部分无效：筹码特征降级为 0，并输出告警（不阻断）
  - `support_price_max_density` 或 `ms_vs_max_logz` 无法解析：降级为 None
  - 14:30 后仍未成交（SUBMITTED）：撤单
  - 下单失败/锁资失败：进入冻结模式
- **关键实现**
  - 前提：`position/scale_prerequisites.py`
  - 信号：`position/scale_signal.py` + `integrations/scale_features.py`
  - 执行：`position/scale_executor.py`

### 第9层：盘后 Phase2 扫描（生成下一交易日 `pending_entries`）

- **触发条件**
  - 主循环在 15:01 后退出盘中循环，进入 `post_close`
- **筛选条件**
  - 盘后会重建候选池，并在启用 watch_auto 时按阈值过滤（同盘前）
  - 质量门控接口 `_allow_phase2_candidate` 目前为“全部放行”（字段预留，但尚未落地）
- **触发条件（Phase2 内部）**
  - 计算 5 个信号并打分；若 `score>=0.45` 且 `diversity_gate` 成立 → 生成 `SignalFired` → 写入 `PendingEntry(status=PENDING_TRIAL)`
  - 强信号：`score>=0.70` 会缩短有效期（详见 `entry/phase2.py`）
- **信号后置门禁（当前正式默认开启）**
  - **确认后禁再入场**：若该 ETF 当前已有持仓，且仓位状态属于 `S2_BASE / S3_SCALED / S4_FULL / S5_REDUCED`，则即使当日再次出现 Phase2 信号，也不会再写入新的 `PendingEntry`
  - **高位追买拦截**：使用 `all_signals + 60天 + 15%` 口径；若当前信号相对窗口内首个有效信号的参考价已上涨超过 `15%`，则阻断该次追买
  - **首信号记忆**：在 `all_signals` 模式下，即使某次信号后续被高位拦截，也仍会保留首信号记忆，用于约束同窗口内后续追买
- **降级条件**
  - 微观字段缺失：`S_micro` 默认记为 0（也可用测试参数 `--phase2-s-micro-missing` 固定一个值以便调试全链路）
  - 任一异常：记录错误并继续
- **关键实现**
  - Phase2：`entry/phase2.py` + `entry/scoring.py`
  - 写入 pending：`entry/entry_fsm.py`
  - 调用点：`strategy_runner.py` 的 `_post_close`

## 4. 横切机制（强烈建议理解）

### 4.1 数据质量门控（DataQuality）

- L1 快照时间戳滞后 > 15 秒 → `STALE`（`core/constants.py` + `core/adapters/data_adapter.py`）
- 被门控阻断的动作（遇到 STALE/MISSING/UNAVAILABLE 会直接拒绝）：
  - 入场确认、加仓确认、T0 信号、Layer1 触发检查、救生衣回补检查等（`core/validators.py`）
- 结果：**宁可不动，也不在数据不新鲜时做关键交易动作**，并输出显式降级信息

### 4.2 “可降级但必须可见”

- 统一通过 `warn_once / degrade_once / alert_once` 输出提示，避免静默回退
- 开关：`QT_STRICT_WARNINGS=1` 可把 INFO/DEBUG 的降级提示提升到 WARNING（便于调试零告警运行）

### 4.3 状态持久化与重启恢复

- **状态文件是什么**
  - 默认是 `data/state/portfolio.json`，也可通过 `--state-path` 指向独立文件
  - 这不是“买卖点图”或普通日志，而是策略本地状态账本
- **会保存哪些关键状态**
  - `positions`：每只 ETF 的状态机位置、持仓数量、成本、分层仓位、`pending_sell_locked`
  - `pending_entries`：待确认入场单
  - `exit_order_intents`：卖单发出后的本地执行语义，用于重启后恢复 `FULL_EXIT / LAYER2_REDUCE`
  - `phase2_high_chase_signals`：高位追买拦截的首信号记忆
- **启动恢复做了什么**
  - `PositionFSM / EntryFSM / ExitFSM` 会先恢复本地状态
  - 随后 `StrategyRunner` 会在启动期与运行期都尝试回放 `exit_order_intents`
  - 若 broker 持仓尚未对齐，会主动 `confirm_order()` 再复查；若仍异常、订单被拒、字段非法或对账失败，都会输出显式 `alert_once`
- **意义**
  - 避免“重启后丢失卖单语义”“延迟成交后本地状态错乱”“S0 但券商仍有仓位”这类问题
  - 模拟盘、实盘、不同账户最好各用一份独立状态文件，避免互相污染

### 4.4 适配器差异（xt / gui）

- `DataAdapter` 默认是 `xtquant.xtdata`（没有则无法运行主循环）
- `TradingAdapter` 两种：
  - `xt`：XtQuant 交易适配器
  - `gui`：easytrader GUI 实盘适配器（存在**操作次数预算**与**持仓查询节流**：10 秒内重复查询会抛异常 → 主循环会记录错误并继续）

### 4.5 组合熔断（Circuit Breaker）

- 触发：
  - 盘中软冻结：`nav_estimate <= HWM * 0.92`
  - 盘中硬触发：`nav_estimate <= HWM * 0.90`
- 影响：冻结新开仓/加仓/T0 买入等；仍允许止损、减仓、T0 卖出
- 关键实现：`position/circuit_breaker.py` + `position/position_fsm.py`

## 5. 关键阈值速查（最常看的那几行）
